Inline completion, чат, vibe coding

Автодополнение

  • Подсказки в реальном времени в IDE
  • Контекст: текущий файл/вкладки
  • Лучший «вход» в AI-assisted coding

GitHub Copilot, Supermaven, Cursor Tab, Codeium

Чат

  • AI без доступа к файлам/терминалу
  • Контекст задаётся вручную
  • Минимум риска, но медленный цикл

ChatGPT, Claude.ai, встроенные чаты IDE

Vibe coding

  • Максимальная скорость для MVP/демо
  • Код принимается «по ощущению»
  • Требует жёсткого ревью перед продом

Cursor Composer, Bolt, v0, Lovable

Ключевой риск vibe coding: если не понимаете код, вы не сможете его безопасно развивать.

Агентский режим: Think → Act → Observe

THINK

Анализ задачи и плана

→

ACT

Вызов инструментов

→

OBSERVE

Анализ результата

→

DECIDE

Продолжить или завершить

Сильная сторона

Агент опирается на факты из файлов/терминала, а не только на текст запроса.

Ограничение

Без правил, прав доступа и human-in-the-loop автономность превращается в риск.

Мультиагенты и background agents

Мультиагентная оркестрация

  • Orchestrator распределяет подзадачи
  • Специализированные субагенты (код, ревью, тесты)
  • Параллельная работа по модулям

Примеры: RooCode Orchestrator, субагенты Claude Code, Devin.

Background / async агенты

  • Триггеры: Issue/PR comments, CI/CD, Slack
  • Результат: PR, коммит, отчёт
  • Контроль в основном через ревью результата

Примеры: OpenAI Codex cloud tasks, Copilot Coding Agent.

Раздел 01b

Субагенты

Как один AI-агент запускает других для параллельной работы

Что такое субагент

Суть

  • Отдельный AI-процесс, запущенный основным агентом
  • Своё контекстное окно и набор инструментов
  • Выполняет подзадачу → возвращает результат родителю

Зачем

  • Параллелизм — несколько задач одновременно
  • Изоляция — не засоряет контекст родителя
  • Специализация — своя роль и инструменты
  • Масштаб — задачи, что не влезут в одну сессию

Как работают субагенты

Родительский агент
│
├─ Субагент 1 (Explore)          ← исследование
│  Инструменты: поиск, чтение
│  Результат: "Нашёл: модуль X в src/lib/"
│
├─ Субагент 2 (General-purpose)  ← реализация
│  Инструменты: чтение, запись, терминал
│  Результат: "Готово, изменил 3 файла"
│
├─ Субагент 3 (Code-reviewer)    ← ревью
│  Инструменты: чтение, поиск
│  Результат: "Замечания: 1 баг, 2 стиль"
│
└─ Родитель собирает и отвечает пользователю

Субагенты 1 и 2 могут работать параллельно, если задачи независимы.

Типы субагентов (Claude Code)

Explore

Быстрый поиск по коду. Только чтение, без записи. Идеален для исследования большой кодовой базы.

Plan

Проектирование архитектуры. Только чтение. Возвращает план реализации.

General-purpose

Полный набор инструментов. Может читать, писать, запускать команды. Для реализации.

Code-reviewer

Ревью написанного кода с чистым контекстом. Объективнее, чем автор.

Worktree-изоляция: субагент может работать в отдельной копии репозитория, не трогая основную ветку.

Субагенты в RooCode: Boomerang Tasks

Orchestrator Mode
│
├─→ Architect Mode
│   "Спроектируй API для экспорта"
│   ← Возвращает: схему API и план файлов
│
├─→ Code Mode
│   "Реализуй API по этой схеме"
│   ← Возвращает: "Создал 3 файла"
│
└─→ Code Mode
    "Напиши тесты для нового API"
    ← Возвращает: "5 тестов, все проходят"

Каждая задача выполняется в режиме, подходящем для неё. Результат автоматически возвращается оркестратору.

Когда использовать субагентов

Да — используй

  • Задача разбивается на независимые части
  • Нужно исследовать большую кодовую базу
  • Изменения в нескольких несвязанных модулях
  • Контекстное окно родителя заполнено
  • Нужен объективный ревью

Нет — не нужно

  • Задача решается за 2-3 шага
  • Изменения в одном файле
  • Субагенты будут редактировать одни файлы
  • Задача требует полного контекста диалога

Параллельные субагенты не должны редактировать одни и те же файлы — будут конфликты. Разделяйте по модулям.

Раздел 02

Ключевые понятия

Skills, Rules, Tools, MCP, Hooks, Plugins

Skills: модульные инструкции по требованию

УровеньКогдаСтоимость контекста
DiscoveryВсегда на старте~100 токенов на skill
InstructionsПри активации skillПолный текст SKILL.md
ResourcesПо необходимостиФайлы/скрипты без загрузки в discovery

Skills экономят контекст, потому что не активны постоянно, в отличие от rules.

Когда skills становится много

Проблемы при 50+ skills

  • Деградация выбора нужного skill
  • Конфликты похожих descriptions
  • Рост базового контекста на каждый запрос

Что помогает

  • Специфичные descriptions и trigger-фразы
  • Разделение по режимам (code/architect)
  • Качество важнее количества

Практика: 10 качественных skills обычно эффективнее, чем 50 поверхностных.

Реальный Skill: write-e2e-test

---
name: write-e2e-test
description: Use when writing or modifying E2E tests — enforces strict test quality standards.
---

# Writing E2E Tests
1. Read tests in e2e/tests/, page code in src/pages/, i18n in public/locales/en/
2. Prefer accessible selectors: getByRole/getByText
3. Every test must assert concrete content/state
4. NEVER use waitForTimeout
5. Add negative scenarios and run eslint + playwright

Рекомендация из практики: добавляйте примеры селекторов и текстов из реального кода проекта.

Rules: рамки поведения, активные всегда

Уровни правил

  • Глобальные
  • Проектные
  • Режимные
  • Условные по путям файлов

Зачем нужны

  • Безопасные границы поведения агента
  • Единый стиль и архитектура
  • Предсказуемость для всей команды
# .roo/rules-code/security.md
- NEVER read or modify `.env*`
- NEVER run `rm -rf`, `DROP TABLE`, `TRUNCATE`
- Ask user confirmation before deleting files

Rules vs Skills: в чём разница

RulesSkills
Когда активныВсегдаOn-demand
РольОграничения/стандартыПроцедуры/знания
КонтекстПостоянный расходТолько при активации

Rules задают рамки «что можно/нельзя», Skills описывают «как выполнять конкретную задачу».

Tools: из текста в действие

Core

  • Read / Write
  • Edit / MultiEdit
  • Search / Grep

Execution

  • Bash / Run
  • Browser
  • Notebook

Meta

  • Skill loader
  • Permission prompts
  • JSON schema
  1. Search находит файл формы
  2. Read читает текущую логику
  3. Edit меняет валидацию
  4. Bash запускает тесты и линт
  5. git diff показывает итог пользователю

MCP: стандарт интеграций

Как работает

  • MCP-сервер отдаёт набор tools
  • Агент вызывает tool через стандартный интерфейс
  • Один агент может подключать много серверов

Почему важно

  • Переносимость между клиентами
  • Меньше vendor lock-in
  • Быстрое расширение возможностей команды

Типичные MCP-серверы: GitHub, Database, Slack/Jira/Notion, Docker, Web scraping.

Hooks: детерминированные триггеры

СобытиеКогдаПример
PreToolUseПеред инструментомБлокировка опасных команд
PostToolUseПосле инструментаАвтоформатирование
SessionStart / StopНачало / конецЗагрузка контекста, уведомление
PermissionRequestЗапрос доступаПолитика approve/deny

Command

Shell-команда с JSON через stdin/stdout.

Prompt

Однопроходная оценка через LLM.

Agent

Запуск субагента для глубокой проверки.

Hooks: практические примеры

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "npx prettier --write $FILE_PATH"
      }]
    }]
  }
}
# PreToolUse hook
if "rm -rf" in cmd:
    print("BLOCKED: dangerous deletion", file=sys.stderr)
    sys.exit(2)

Plugins (OpenCode) и эмуляция hooks

OpenCode plugins

  • Полноценный JS/TS-код
  • События: before/after, session, permission
  • Высокая гибкость и интеграция с npm

Если hooks нет

  • Создать кастомный tool (lint_check)
  • Зафиксировать протокол в rules
  • Продублировать требование в mode instructions

Это не 100% гарантия, но рабочая стратегия для инструментов без нативных hooks.

Сборка: Skills + Rules + Tools + Hooks

Skills

Что: процедура конкретной задачи.

Когда: по требованию.

Пример: write-e2e-test.

Rules

Что: ограничения и стандарты.

Когда: всегда активны.

Пример: запрет rm -rf, .env*.

Tools

Что: реальные действия в проекте.

Когда: в каждом шаге цикла.

Пример: Read → Edit → Bash → diff.

Hooks

Что: гарантированные срабатывания.

Когда: lifecycle-события.

Пример: автоформат после Edit/Write.

Формула: Skills = как делать, Rules = что нельзя, Tools = чем делать, Hooks = что произойдёт гарантированно.

Раздел 03

Что хранить в репозитории

Командные настройки в Git, персональные и секреты — только локально

Коммитить vs не коммитить

Коммитить ✓

  • AGENTS.md, CLAUDE.md
  • .clinerules/, .roo/rules*/
  • Проектные skills/ и commands/
  • Проектные hooks/config без секретов

Оставлять локально ✗

  • .env*, API keys, credentials
  • Глобальные пользовательские предпочтения
  • *.local.json и локальные overrides
  • Секретные MCP-конфиги

Добавьте в .gitignore: *.local.json, .env*, любые файлы с API-ключами.

Рекомендуемая структура проекта

my-project/ ├── AGENTS.md ├── CLAUDE.md ├── .clinerules/ ├── .cline/skills/ ├── .roo/ │ ├── rules/ │ ├── rules-code/ │ ├── rules-architect/ │ └── skills/ ├── .roomodes ├── .opencode/ │ ├── commands/ │ └── agents/ ├── .claude/ │ ├── skills/ │ └── settings.json └── src/...

Минимум: закоммитьте стандарты и процедуры, чтобы поведение агента было одинаковым у всей команды.

Раздел 04

Рекомендации по работе с агентами

Постановка задач, контроль, командные процессы и антипаттерны

Формулировка задач + контроль

Формулировка задачи

  • Конкретный target-файл и ожидаемый результат
  • Явные ограничения: что не менять
  • Большие задачи делить на подзадачи
  • Для сложных задач: plan first

Контроль и ревью

  • Всегда смотреть git diff
  • Запускать тесты после каждого значимого шага
  • Коммитить checkpoint перед автономным запуском
  • Auto-approve только для безопасной рутины

Организация работы + чего избегать

Организация

  • Один агент = одна задача
  • Feature-branch + ревью через PR
  • Rules и Skills с первого дня
  • MCP для интеграции с командными сервисами

Антипаттерны

  • Слепое принятие изменений
  • Работа без Git и без отката
  • Слишком длинные сессии
  • Секреты в конфиге агента

Итоги

Ключевые выводы

1

Шесть режимов AI формируют спектр автономии, а не бинарный выбор.

2
