## 1. Режимы работы с AI

Прежде чем говорить об инструментах, важно разобраться в режимах взаимодействия разработчика с AI. Каждый предполагает разную степень автономии модели, разный уровень контроля со стороны человека и, как следствие, разные риски и возможности. Режимы выстраиваются в спектр — от минимального вмешательства AI до полной автономии.

### 1.1. Автодополнение (Inline Completion)

Самый незаметный и массовый режим. AI подсказывает следующую строку, блок или целую функцию прямо в редакторе. Разработчик принимает подсказку нажатием Tab или отклоняет, продолжая набирать код. Модель работает проактивно — не нужно ничего спрашивать.

- AI предлагает код **в реальном времени** по мере набора
- Минимальное трение: Tab для принятия, Esc для отклонения
- Контекст берётся из **текущего файла и открытых вкладок** автоматически
- Разработчик полностью контролирует, что принять
- Не подходит для сложных задач — работает на уровне строк и небольших блоков

**Инструменты:** GitHub Copilot, Supermaven, Continue, Cursor Tab, Codeium, TabNine.

> *Автодополнение — это «точка входа» в AI-assisted development. Большинство разработчиков начинают именно с него и часто даже не осознают, что уже используют AI в работе.*

### 1.2. Чат-режим

Самый привычный и консервативный подход. Разработчик общается с моделью через текстовый интерфейс (ChatGPT, Claude.ai, встроенный чат в IDE). Модель отвечает текстом, код выдаёт в виде блоков, которые разработчик вручную копирует и вставляет в проект.

- Модель **не имеет доступа** к файловой системе, терминалу или редактору
- Весь контекст задаётся вручную — через copy-paste или прикрепление файлов
- Разработчик — **единственный исполнитель**: AI только советует
- Риск ошибки минимален, но скорость и масштаб ограничены

> *Типичные задачи: объяснение кода, написание функций по описанию, генерация тестов, ответы на технические вопросы.*

### 1.3. Вайбкодинг (Vibe Coding)

Термин, популяризированный Андреем Карпатым в 2025 году. Суть: разработчик описывает высокоуровневое желание на естественном языке, а AI генерирует весь код целиком. Разработчик принимает результат «по ощущению» (vibe), не вникая глубоко в детали реализации.

- Фокус на **скорости получения результата**, а не на понимании кода
- Часто используется с Cursor Composer, Bolt, v0, Lovable и подобными
- Хорошо работает для **прототипов и одноразовых проектов**
- Опасен для продакшена: технический долг накапливается незаметно
- Разработчик может потерять контроль над архитектурой

> *Ключевой риск: если разработчик не понимает сгенерированный код, он не может его поддерживать, отлаживать и развивать.*

### 1.4. Агентский режим

Наиболее продвинутый режим. AI получает доступ к инструментам — файловой системе, терминалу, браузеру, LSP, внешним API — и работает в цикле «думаю → действую → наблюдаю» (Think → Act → Observe). Агент может самостоятельно планировать, выполнять код, анализировать ошибки и итеративно улучшать результат.

- AI **автономно** читает/пишет файлы, запускает команды, анализирует вывод
- Работает в цикле (agentic loop): планирование → исполнение → анализ результата
- Human-in-the-loop: пользователь подтверждает критические действия
- Масштаб задач: от одного файла до рефакторинга целых модулей
- Требует **правильной настройки контекста** и понимания работы инструмента

### 1.5. Мультиагентная оркестрация

Следующий уровень — не один агент, а несколько, каждый со своей специализацией, координируемых мета-агентом (Orchestrator). Один агент планирует, другой пишет код, третий проводит ревью, четвёртый запускает тесты. Задача декомпозируется на подзадачи, которые выполняются параллельно или последовательно.

- **Orchestrator** распределяет работу между специализированными агентами
- Каждый агент имеет свой набор tools, ограничения и системный промпт
- Субагенты могут работать **параллельно** (разные файлы/модули одновременно)
- Результаты агрегируются, конфликты разрешаются
- Требует продуманной **декомпозиции задач** и настройки режимов

**Инструменты:** RooCode Orchestrator + Boomerang Tasks, Claude Code субагенты, Oh-My-OpenCode (плагин для OpenCode), Devin.

> *Мультиагентная оркестрация — это переход от «AI как помощник» к «AI как команда». Эффективна для крупных задач, но требует значительно больше настройки и контроля.*

### 1.6. Background / Async Agents (автономные фоновые агенты)

Самый автономный режим. Агент работает в фоне — без прямого участия разработчика. Триггером может быть комментарий в GitHub Issue, PR, расписание CI/CD или команда в Slack. Результат — готовый PR, коммит или отчёт. Разработчик подключается только на этапе ревью.

- Агент работает **асинхронно**: запустил и забыл до результата
- Триггеры: GitHub comments, CI/CD pipeline, Slack, webhook
- Результат: Pull Request, коммит, отчёт, Issue
- Разработчик контролирует через **ревью PR**, а не одобрение каждого действия
- Требует высокого уровня доверия к агенту и хорошего покрытия тестами

**Инструменты:** OpenAI Codex (cloud tasks), GitHub Copilot Coding Agent, RooCode Cloud Agents, OpenCode GitHub Actions (`/opencode` в комментариях), Claude Code в CI/CD.

> *Background Agents — это будущее, которое уже наступает. Идеально для рутины: исправление багов по Issue, обновление зависимостей, генерация тестов. Но для архитектурных решений человек по-прежнему незаменим.*

### 1.7. Сравнительная таблица режимов

| Параметр | Автодополнение | Чат | Вайбкодинг | Агентский | Мультиагентный | Background |
|---|---|---|---|---|---|---|
| Контроль разработчика | Полный (Tab/Esc) | Полный | Минимальный | Настраиваемый | Настраиваемый | Только ревью |
| Автономия AI | Минимальная | Нулевая | Высокая | Высокая, контролируемая | Очень высокая | Полная |
| Доступ к файлам | Текущий файл | Нет | Через инструмент | Прямой | Прямой | Прямой |
| Доступ к терминалу | Нет | Нет | Ограниченный | Полный | Полный | Полный |
| Скорость | Мгновенная | Низкая | Очень высокая | Высокая | Высокая | Асинхронно |
| Риск ошибок | Минимальный | Низкий | Высокий | Средний | Средний | Средний-высокий |
| Подходит для | Рутинный код | Точечные задачи | Прототипы, MVP | Рабочие задачи | Крупные фичи | Рутина, CI/CD |

## 2. Как устроен агентский инструмент изнутри

Несмотря на различия в интерфейсе и конфигурации, все агентские инструменты построены на одной и той же базовой архитектуре. Понимание этой архитектуры помогает эффективнее настраивать и использовать любой из них.

### 2.1. Agentic Loop (цикл агента)

Ядро любого агентского инструмента — это цикл, в котором LLM-модель последовательно планирует, выполняет действия и анализирует результаты:

```
while задача_не_завершена:
    1. THINK  — модель анализирует контекст и планирует действие
    2. ACT    — вызывает инструмент (tool) через tool_use API
    3. OBSERVE — получает результат и добавляет в контекст
    4. DECIDE — решает: продолжить, запросить одобрение или завершить
```

### 2.2. Компоненты системы

- **System Prompt** — начальные инструкции для модели: роль, правила, доступные tools
- **Tool Definitions** — описание доступных инструментов в формате JSON Schema
- **Tool Use API** — механизм вызова инструментов (function calling / tool_use)
- **Context Window** — окно контекста модели, в которое помещается история диалога и результаты
- **Permission System** — механизм одобрения действий пользователем (human-in-the-loop)
- **File System Access** — чтение/запись файлов проекта
- **Shell Access** — выполнение команд в терминале
- **LSP Integration** — диагностика, автодополнение, навигация по коду
- **MCP Servers** — внешние инструменты, подключаемые через стандартный протокол

### 2.3. Схема взаимодействия

```
┌─────────────────────────────────────────┐
│  User prompt (+ System Prompt + Rules)  │
└──────────────────┬──────────────────────┘
                   │
            ┌──────┴───────┐
            │  LLM Model   │
            └──────┬───────┘
                   │ tool_use call
         ┌────────┴─────────┐
         │   Tool Router    │
         └─┬────┬────┬────┬─┘
           │    │    │    │
         Files Shell LSP  MCP
```

### 2.4. Принципиальное отличие от чата

Ключевое отличие агентского режима — это наличие **инструментов** (tools) и **цикла обратной связи**. В чат-режиме модель генерирует текст одним проходом. В агентском режиме модель может совершить десятки вызовов инструментов в рамках одной задачи, каждый раз получая новую информацию и корректируя план.

## 3. Ключевые понятия: Skills, Rules, Tools, MCP, Hooks

Эти пять концепций составляют «алфавит» настройки любого AI-агента. Они реализованы в каждом инструменте по-разному, но суть везде одна.

### 3.1. Skills (навыки)

Skills — это модульные наборы инструкций, которые агент загружает **по требованию** для выполнения конкретных задач. В отличие от Rules (которые всегда активны), Skills подключаются on-demand и не потребляют контекст, когда не нужны.

**Формат:** папка с файлом `SKILL.md` (YAML frontmatter + Markdown-инструкции) и опциональными ресурсами (скрипты, шаблоны, документация).

**Прогрессивная загрузка:**

| Уровень | Когда | Стоимость |
|---|---|---|
| Discovery | Всегда (при старте) | ~100 токенов на skill (только name + description) |
| Instructions | При активации skill | <5k токенов (полный SKILL.md) |
| Resources | По необходимости | Без ограничений (файлы через read_file, скрипты) |

**Пример SKILL.md:**

```yaml
---
name: deploy
description: Deploy the application to production. Use when deploying, updating infrastructure, or managing releases.
---

# Deploy

1. Run the test suite
2. Build the application
3. Push to the deployment target

For rollback instructions, see [rollback.md](docs/rollback.md).
```

**Стандарт Agent Skills** (agentskills.io) — открытый формат, поддерживаемый Claude Code, OpenCode, RooCode, OpenAI Codex CLI.

**Реальный пример Skill (E2E тесты):**

```yaml
---
name: write-e2e-test
description: Use when writing or modifying E2E tests — enforces strict test quality standards.
---

# Test Writing Protocol
1. Before writing: read `e2e/tests/`, page source in `src/pages/`, i18n keys in `public/locales/en/`
2. Selectors: prefer `getByRole/getByText`; avoid fragile class selectors
3. Assertions: verify concrete content/state, not just URL changes
4. Waiting: never use `waitForTimeout`; use explicit `expect`/`waitForResponse`
5. For each form add negative scenario: invalid input, empty required fields, failed submit stays on same URL
6. After writing run: `npx eslint --ext .ts e2e/tests/<file>.spec.ts` and `npx playwright test e2e/tests/<file>.spec.ts`
```

Этот skill удобно подключать по триггерам типа: «добавь e2e тест», «обнови e2e после изменения формы», «проверь негативные сценарии логина».

**Рекомендация:** включайте в skill примеры из реального кода проекта (реальные селекторы, тексты из i18n, маршруты, поля форм), а не абстрактные шаблоны. Это резко снижает хрупкость E2E тестов.

**Масштабирование: что происходит, когда skills становится много**

Progressive disclosure смягчает проблему, но не устраняет полностью. На этапе Discovery агент видит name + description каждого skill — ~100 токенов на штуку. При 50 skills это ~5000 токенов в системном промпте *при каждом запросе*, даже когда ни один skill не нужен.

Реальные проблемы при большом количестве skills:

- **Деградация выбора.** При 5–10 skills модель отлично матчит запрос с нужным skill. При 50+ с похожими description — может выбрать не тот или вообще не распознать совпадение. Классическая проблема «needle in a haystack» для метаданных.
- **Конфликты описаний.** Два skill с пересекающимися description могут активироваться одновременно или соперничать. Например, `api-testing` и `integration-testing` при запросе «написать тесты для API».
- **Рост базового потребления контекста.** 100 токенов × 50 skills = 5000 токенов, которые оплачиваются при каждом запросе. При контекстном окне в 200k это немного, но при интенсивной работе набегает.

Рекомендации:

- **Специфичные description** — непересекающиеся, с конкретными trigger-фразами
- **Mode-specific skills** — в RooCode: `.roo/skills-code/` vs `.roo/skills-architect/`, в каждом режиме видны только релевантные
- **Отключение неиспользуемых** — через UI toggle (Cline) или удаление из директории
- **Качество > количество** — 10 хорошо написанных skills лучше 50 поверхностных
- **Группировка** — лучше один skill `database-migrations` с секциями, чем отдельные `create-migration`, `run-migration`, `rollback-migration`

### 3.2. Rules (правила)

Rules определяют, как агент должен себя вести: какие файлы можно менять, какие команды выполнять, какой стиль кода использовать. Правила **всегда активны** и встраиваются в системный промпт.

**Уровни правил:**

- **Глобальные** — применяются ко всем проектам (`~/.roo/rules/`, `~/.config/opencode/AGENTS.md`)
- **Проектные** — специфичны для репозитория (`.clinerules/`, `.roo/rules/`, `AGENTS.md`)
- **Режимные** — для конкретного режима (`.roo/rules-code/`, `.roo/rules-architect/`)
- **Условные** — активируются по путям файлов (Cline: YAML frontmatter с `paths`)

**Пример:**

```markdown
# .clinerules/coding-style.md
---
paths:
  - "src/components/**"
  - "src/hooks/**"
---
## React Guidelines
- Используй functional components с React hooks
- Извлекай переиспользуемую логику в custom hooks
- Keep components focused on a single responsibility
```

**Реальный пример Rule (безопасность):**

```markdown
# .roo/rules-code/security.md

## Security Constraints
- NEVER read or modify `.env*` files
- NEVER run destructive commands: `rm -rf`, `DROP TABLE`, `TRUNCATE`
- Ask for explicit user confirmation before deleting any file
- Mask secrets in logs and outputs (`token`, `password`, `api_key`)
```

Такое правило снижает риск критичных ошибок даже при длинных автономных сессиях.

### 3.3. Rules vs Skills: в чём разница

| | Rules | Skills |
|---|---|---|
| Когда активны | Всегда | По требованию |
| Назначение | Ограничения и стандарты (что делать / не делать) | Знания и процедуры (как делать конкретные вещи) |
| Потребление контекста | Постоянное | Только при активации |
| Примеры | Стиль кода, запрещённые операции, архитектура | Деплой, генерация тестов, ревью, миграции |

Rules задают рамки, Skills наполняют их содержанием. Оба механизма дополняют друг друга.

### 3.4. Tools (инструменты)

Tools — это то, что превращает LLM из генератора текста в агента. Каждый инструмент описан в формате JSON Schema и доступен модели через механизм tool_use (function calling).

**Типичный набор встроенных tools:**

- **Read / Write** — чтение и запись файлов
- **Edit / MultiEdit** — точечное редактирование участков кода
- **Bash / Run** — выполнение команд в терминале
- **Search / Grep / Glob** — поиск по файлам и содержимому
- **Browser** — управление headless-браузером
- **Notebook** — работа с Jupyter-ноутбуками
- **Skill** — загрузка и активация Skills

**Реальный пример применения Tools (фикс бага):**

Задача: «Исправить валидацию email в форме регистрации».

1. `Search/Grep` — найти компонент формы: `src/components/RegisterForm.tsx`
2. `Read` — прочитать код компонента и текущие проверки
3. `Edit` — заменить regex/валидацию и текст ошибки
4. `Bash` — запустить `npm test -- RegisterForm` и `npm run lint`
5. `Read`/`Git diff` — показать итоговые изменения для ревью

Это типичный agentic loop: найти → изменить → проверить → показать diff.

### 3.5. MCP (Model Context Protocol)

MCP — это открытый протокол от Anthropic для стандартизации подключения внешних инструментов к LLM-агентам. Он решает ключевую проблему: каждый инструмент ранее требовал кастомной интеграции, MCP делает это универсальным.

**Как работает:**

- MCP-сервер — это процесс, предоставляющий набор tools через стандартный API
- Агент подключается к MCP-серверу и получает описание доступных tools
- При необходимости агент вызывает tool через сервер и получает результат
- Один агент может быть подключён к множеству MCP-серверов одновременно

**Примеры MCP-серверов:**

- **GitHub** — работа с Issues, PR, Actions
- **Database** — запросы к БД (PostgreSQL, SQLite...)
- **Slack / Jira / Notion** — интеграция с рабочими инструментами
- **Web scraping** — поиск и извлечение данных из веба
- **Docker** — управление контейнерами
- **Filesystem** — расширенные операции с файлами

**Почему MCP важен:**

- Стандартизация — один MCP-сервер работает с любым совместимым агентом
- Переносимость — написали MCP-сервер для Cline, он работает и с OpenCode
- Расширяемость — легко добавлять новые возможности агенту
- Безопасность — MCP-сервер может иметь свою систему авторизации и ограничений

### 3.6. Hooks (хуки)

Hooks — это детерминированные триггеры, которые запускают пользовательские скрипты в определённые моменты жизненного цикла агента. В отличие от промптов (которые модель может проигнорировать), hooks срабатывают **гарантированно** при каждом событии.

> **Промпты — это предложения. Hooks — это гарантии.**

**Типичные события:**

| Событие | Когда | Пример использования |
|---|---|---|
| PreToolUse | Перед вызовом инструмента | Блокировка опасных команд (rm -rf) |
| PostToolUse | После вызова инструмента | Автоформатирование кода (prettier) |
| SessionStart | Начало сессии | Загрузка контекста (issues, git status) |
| UserPromptSubmit | Отправка промпта | Валидация, инъекция контекста |
| PermissionRequest | Запрос разрешения | Автоматическое approve/deny |
| Stop | Завершение ответа | Коммит, уведомление, cleanup |

**Три типа обработчиков (Claude Code):**

- **Command** — запуск shell-команды. Получает JSON через stdin, управляет через exit code
- **Prompt** — отправка промпта модели для однопроходной оценки
- **Agent** — запуск субагента с доступом к инструментам для глубокой проверки

**Пример: автоформатирование после записи файла:**

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "npx prettier --write $FILE_PATH"
      }]
    }]
  }
}
```

**Пример: блокировка опасных команд:**

```python
# PreToolUse hook
import json, sys
data = json.load(sys.stdin)
cmd = data.get("tool_input", {}).get("command", "")
if "rm -rf" in cmd:
    print("BLOCKED: dangerous deletion", file=sys.stderr)
    sys.exit(2)  # exit code 2 = deny action
```

**Hooks в разных инструментах:**

- **Claude Code** — наиболее зрелая реализация: 12 событий, 3 типа обработчиков, модификация tool input, JSON-контроль решений
- **OpenCode** — система плагинов с событиями (tool.execute.before/after, session.*, file.*, shell.env)
- **Cursor** — поддержка hooks с v1.7: beforeShellExecution, afterFileEdit, stop
- **Cline** — система hooks (экспериментальная)

### 3.7. Plugins (на примере OpenCode)

В OpenCode роль hooks выполняет система плагинов — JS/TS-модули, которые подписываются на события жизненного цикла агента. По сути это те же hooks, но реализованные как код, а не как конфиг.

**Подключение:**

- Локальные: `.opencode/plugins/my-plugin.ts` (проект) или `~/.config/opencode/plugins/` (глобально)
- npm-пакеты: `"plugin": ["opencode-wakatime"]` в `opencode.json`

**Структура плагина:**

```typescript
// .opencode/plugins/auto-format.ts
export const AutoFormatPlugin = async ({ project, client, $ }) => {
  return {
    // Перехват ПОСЛЕ записи файла — аналог PostToolUse
    "tool.execute.after": async (input, output) => {
      if (input.tool === "write" || input.tool === "edit") {
        const filePath = output.args?.filePath || ""
        if (filePath.endsWith(".ts") || filePath.endsWith(".tsx")) {
          await $`npx prettier --write ${filePath}`
        }
      }
    }
  }
}
```

**Основные события (аналогия с Claude Code hooks):**

| OpenCode Plugin Event | Аналог в Claude Code | Что делает |
|---|---|---|
| `tool.execute.before` | PreToolUse | Перехват перед вызовом инструмента. Можно модифицировать аргументы или заблокировать (`throw Error`) |
| `tool.execute.after` | PostToolUse | Действие после инструмента: форматирование, логирование, тесты |
| `session.created` | SessionStart | Инициализация при старте сессии |
| `session.idle` | Stop | Агент завершил работу — уведомление, коммит |
| `session.error` | — | Обработка ошибок сессии |
| `file.edited` | — | Реакция на изменение файла |
| `permission.asked` | PermissionRequest | Перехват запросов разрешений |
| `shell.env` | — | Инъекция переменных окружения в shell |

**Пример: защита .env файлов от чтения:**

```typescript
// .opencode/plugins/env-protection.ts
export const EnvProtection = async ({ $ }) => {
  return {
    "tool.execute.before": async (input, output) => {
      if (input.tool === "read" && output.args.filePath.includes(".env")) {
        throw new Error("Reading .env files is blocked for security")
      }
    }
  }
}
```

**Пример: уведомление при завершении задачи (macOS):**

```typescript
// .opencode/plugins/notify.ts
export const NotifyPlugin = async ({ $ }) => {
  return {
    event: async ({ event }) => {
      if (event.type === "session.idle") {
        await $`osascript -e 'display notification "Done!" with title "OpenCode"'`
      }
    }
  }
}
```

**Ключевое отличие от Claude Code hooks:** плагины OpenCode — это полноценный код (не shell-скрипты через stdin/stdout), с доступом к SDK, shell API и npm-пакетам. Это даёт больше гибкости, но требует базовых навыков JS/TS.

### 3.8. Когда нет hooks: эмуляция через правила и кастомные инструменты

Не все инструменты имеют полноценные hooks. В RooCode, например, нет глобального автоматического триггера «перед любой записью» — агент сам решает, когда вызывать инструмент. Но можно подойти близко через комбинацию **custom tools + строгие правила в инструкциях**.

**Задача:** перед любой записью в репозиторий запускать линтер.

**Шаг 1. Создать MCP-сервер или кастомный инструмент `lint_check`:**

```typescript
// Упрощённый пример MCP-tool
import { exec } from "node:child_process"
import { promisify } from "node:util"
const execAsync = promisify(exec)

// Tool: lint_check
async function execute({ scope }) {
  const target = scope || "."
  try {
    const { stdout } = await execAsync(`npm run lint -- ${target}`)
    return `ok: lint passed.\n${stdout}`
  } catch (err) {
    return [
      "fail: lint failed. DO NOT modify code until issues are resolved.",
      err.stdout || "",
      err.stderr || ""
    ].join("\n")
  }
}
```

**Шаг 2. В правилах режима прописать протокол:**

```markdown
# .roo/rules-code/lint-before-write.md

## Обязательная проверка перед изменениями

КРИТИЧЕСКИ ВАЖНО: перед ЛЮБЫМ изменением файлов кода (edit, write)
ты ОБЯЗАН сначала вызвать инструмент `lint_check` с указанием scope.

- Если ответ начинается с `ok:` — можно вносить изменения
- Если ответ начинается с `fail:` — ЗАПРЕЩЕНО вносить изменения.
  Вместо этого помоги пользователю исправить ошибки линта
  и запусти `lint_check` снова.

Это правило применяется к файлам в `src/**`, но НЕ к `*.md`, `*.test.*`.
```

**Шаг 3. Настроить права в кастомном режиме (.roomodes):**

```json
{
  "customModes": [{
    "slug": "safe-code",
    "name": "Safe Code",
    "roleDefinition": "Developer with mandatory lint checks before edits",
    "groups": [
      "read",
      ["edit", { "fileRegex": "\\.(ts|tsx|js|jsx)$" }],
      "command",
      "mcp"
    ],
    "customInstructions": "Always call lint_check before any file modification."
  }]
}
```

**Важный нюанс:** это не настоящий системный хук — агент *может* в теории пропустить проверку. Но на практике, если правило сформулировано чётко и повторяется в нескольких местах (rules + mode instructions), модель следует ему стабильно. Для критичных сценариев, где нужна 100% гарантия, используйте инструменты с настоящими hooks (Claude Code, OpenCode).

## 4. Что хранить в репозитории проекта

Один из самых практических вопросов: какие конфигурации AI-агентов должны быть частью проекта (коммитятся в Git), а какие остаются персональными настройками.

### 4.1. Коммитить в репозиторий ✓

Эти файлы полезны всей команде:

- **AGENTS.md** — контекст проекта (открытый стандарт, читают OpenCode, RooCode, Cursor и др.)
- **CLAUDE.md** — инструкции для Claude Code
- **.clinerules/** — директория правил Cline
- **.cline/skills/** — проектные Skills для Cline
- **.claude/skills/** — проектные Skills для Claude Code
- **.claude/settings.json** — проектные hooks и правила Claude Code
- **.roo/rules/** — общие правила RooCode
- **.roo/rules-{mode}/** — правила для конкретных режимов
- **.roo/skills/** — проектные Skills для RooCode
- **.roomodes** — кастомные режимы RooCode (JSON)
- **.opencode/commands/** — проектные команды OpenCode
- **.opencode/agents/** — проектные агенты OpenCode
- **.rooignore** — файлы, скрытые от RooCode

### 4.2. НЕ коммитить (оставить локальным) ✗

- **API-ключи** — всегда в переменных окружения или в ~/.config/
- **Глобальные настройки** — ~/.config/opencode/, ~/.claude/, ~/.roo/ (персональные предпочтения)
- **.claude/settings.local.json** — локальные override для hooks
- **MCP-конфигурации с credentials** — токены, пароли к внешним сервисам
- **Модель по умолчанию** — каждый разработчик может предпочитать свою

### 4.3. Структура проекта (пример)

```
my-project/
├── AGENTS.md                 # контекст (OpenCode, RooCode, Cursor...)
├── CLAUDE.md                 # инструкции для Claude Code
├── .clinerules/              # правила для Cline
│   ├── coding-style.md
│   └── architecture.md
├── .cline/skills/            # Skills для Cline
├── .roo/
│   ├── rules/                # общие правила RooCode
│   ├── rules-code/           # правила для режима Code
│   ├── rules-architect/      # правила для Architect
│   └── skills/               # проектные Skills RooCode
├── .roomodes                 # кастомные режимы RooCode
├── .opencode/
│   ├── commands/             # команды OpenCode
│   └── agents/               # агенты OpenCode
├── .claude/
│   ├── skills/               # проектные Skills (SKILL.md)
│   └── settings.json         # hooks для Claude Code
├── .rooignore                # скрытые файлы для RooCode
├── .gitignore                # включая *.local.json
└── src/...
```

> Добавьте в .gitignore: `*.local.json`, `.env*`, а также любые файлы с API-ключами.

## 5. Рекомендации по работе с AI-агентами

### 5.1. Организация командной работы

- **Один агент — одна задача**: не смешивайте рефакторинг и новые фичи в одной сессии
- Используйте **ветки**: агент работает в feature-branch, ревью через PR
- Настройте **Rules для проекта**: это инвестиция, окупающаяся с первого дня
- Создайте **Skills для повторяющихся задач**: деплой, ревью, генерация тестов
- Используйте **MCP** для интеграции с рабочими инструментами команды

## 6. Итоги и что дальше

### Ключевые выводы

- **Шесть режимов** — от автодополнения до фоновых агентов — образуют спектр автономии. Агентский режим и мультиагентная оркестрация наиболее продуктивны для реальной работы; background agents перспективны для рутины.

- **Skills, Rules, Tools и Hooks** — четыре уровня настройки агента. Rules и Skills коммитятся в репозиторий, Hooks обеспечивают детерминированные гарантии.
- Формат **SKILL.md** и **AGENTS.md** — открытые стандарты, поддерживаемые большинством агентских инструментов.
