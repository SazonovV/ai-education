Режимы работы с AI

Три базовых режима: что выбрать

Автодополнение

  • Мгновенные подсказки строк/блоков
  • Полный контроль у разработчика
  • Лучше всего для рутинного кода

Чат

  • AI без прямого доступа к проекту
  • Контекст передаётся вручную
  • Низкий риск, но медленный цикл

Vibe coding

  • Максимум скорости для прототипа
  • Высокий риск потери контроля
  • Опасно переносить в прод без ревью

Не делегируйте агенту зону, где вы не сможете самостоятельно отладить итоговый код.

Агентский режим: центральный цикл

THINK

Анализ задачи и плана

→

ACT

Вызов tools

→

OBSERVE

Чтение результатов

→

DECIDE

Продолжить или завершить

Сильная сторона

Цикл обратной связи позволяет модели исправляться на основании фактов, а не только текста.

Ограничение

Без правил и permission-границ агент может быть быстрым, но нестабильным и рискованным.

Мультиагентный и background режимы

Мультиагентная оркестрация

  • Orchestrator распределяет подзадачи
  • Параллельная работа по разным файлам
  • Подходит для больших feature-веток

Примеры: RooCode Orchestrator, субагенты Claude Code.

Background / async агенты

  • Триггеры: Issue, PR comment, CI/CD, Slack
  • Результат: PR / коммит / отчёт
  • Контроль в основном на этапе ревью

Примеры: OpenAI Codex tasks, Copilot Coding Agent.

Раздел 02

Как устроен агентский инструмент

Единая архитектура: модель + инструменты + разрешения + цикл

Agentic Loop в псевдокоде

while not done:
    thought = model.plan(context)
    action = model.choose_tool(thought)
    result = run_tool(action)
    context.append(result)

    if needs_user_approval(result):
        ask_human()

    done = model.decide_stop(context)

Ключ

Каждый новый tool_result меняет контекст и следующий шаг модели.

Отличие от чата

Не один проход текста, а серия проверяемых действий с обратной связью.

Компоненты архитектуры агента

Вход

  • System Prompt
  • User Prompt
  • Rules
  • Skills

Ядро

  • LLM
  • Tool Use API
  • Context Window
  • Permission System

Исполнение

  • File system
  • Shell
  • LSP
  • MCP servers

Практический вывод: сбои обычно происходят не в модели, а в неполно описанных правилах/ограничениях.

Отличие от чата в одной схеме

Чат-режим

  • Один ответ текстом
  • Нет доступа к окружению
  • Проверка результата полностью на человеке

Агентский режим

  • Десятки вызовов tools
  • Сбор фактов из проекта/терминала
  • Итеративное улучшение до целевого состояния

Агент эффективен, когда ограничения детерминированы, а критерии завершения ясны.

Раздел 03

Skills, Rules, Tools, MCP, Hooks

Базовый набор для управляемого агента в рабочем проекте

Skills: модульные инструкции по требованию

УровеньКогда загружаетсяНагрузка
DiscoveryНа старте~100 токенов/skill
InstructionsПри активацииТекст SKILL.md
ResourcesПо запросуФайлы/скрипты без лимита в discovery

Практика: лучше 8-12 чётких skills, чем 50 пересекающихся с расплывчатыми descriptions. Для качества добавляйте в skills примеры из реального кода проекта.

Реальный пример Skill: write-e2e-test

---
name: write-e2e-test
description: Use when writing or modifying E2E tests — enforces strict test quality standards.
---

# Writing E2E Tests
1. Read existing tests in `e2e/tests/`, page code in `src/pages/`, i18n in `public/locales/en/`
2. Prefer accessible selectors (`getByRole`, `getByText`); avoid fragile class selectors
3. Every test must assert concrete UI/content result
4. NEVER use `waitForTimeout`; use explicit waits and response checks
5. Add negative scenarios for forms and run eslint + playwright for target spec

Рекомендация: используйте примеры из реального кода (селекторы, тексты, маршруты), а не абстрактные плейсхолдеры.

Rules: рамки поведения, активные всегда

Уровни правил

  • Глобальные
  • Проектные
  • Режимные
  • Условные по путям

Что решают

  • Ограничения на деструктивные действия
  • Единый стиль и архитектурные стандарты
  • Единые ожидания команды от агента
# .roo/rules-code/security.md
- NEVER read or modify `.env*`
- NEVER run `rm -rf`, `DROP TABLE`, `TRUNCATE`
- Ask user confirmation before deleting files

Rules vs Skills

RulesSkills
Когда активныВсегдаПо требованию
ЗачемОграничения и стандартыПроцедуры для конкретных задач
Потребление контекстаПостоянноеТолько при вызове

Rules задают «что допустимо», Skills дают «как это выполнять эффективно».

Tools: из текста в действие

Core tools

  • Read / Write
  • Edit / MultiEdit
  • Search / Grep

Execution

  • Bash / Run
  • Browser
  • Notebook

Meta

  • Skill loader
  • Permission prompts
  • Tool schemas
  1. Search находит файл формы
  2. Read читает текущую логику
  3. Edit меняет валидацию
  4. Bash запускает тесты и линт
  5. git diff показывает итог пользователю

MCP: стандарт интеграций

Что это даёт

  • Единый протокол для внешних tools
  • Переносимость между агентами
  • Быстрое расширение возможностей

Типичные серверы

  • GitHub: PR/Issues/Actions
  • Database: Postgres/SQLite
  • Slack/Jira/Notion

MCP уменьшает vendor lock-in: один сервер можно использовать в разных агентских клиентах.

Hooks: гарантируемые триггеры

СобытиеКогдаПример
PreToolUseПеред инструментомБлокировка опасных команд
PostToolUseПосле инструментаАвтоформатирование
SessionStart / StopНачало / конецЗагрузка контекста, уведомление
PermissionRequestЗапрос доступаПолитика approve/deny

Промпт может быть проигнорирован. Hook по событию срабатывает детерминированно.

Plugins (OpenCode) и эмуляция hooks

OpenCode plugins

  • Полноценный JS/TS-код
  • События: before/after, session, permission
  • Высокая гибкость и интеграция с npm

Если hooks нет

  • Создать кастомный tool (lint_check)
  • Зафиксировать протокол в rules
  • Продублировать требование в mode instructions

Это не абсолютная гарантия, но стабильная рабочая стратегия для инструментов без нативных hooks.

Сборка: Skills + Rules + Tools + Hooks

Skills

Что дают: процедуру выполнения конкретной задачи.

Когда: подключаются по требованию.

Пример: write-e2e-test для E2E сценариев.

Rules

Что дают: рамки и запреты, обязательные для всех запусков.

Когда: активны всегда.

Пример: запрет на rm -rf и чтение .env*.

Tools

Что дают: реальные действия в проекте и окружении.

Когда: в каждом шаге agentic loop.

Пример: Read → Edit → Bash → diff.

Hooks

Что дают: детерминированные срабатывания на события.

Когда: на lifecycle-событиях (before/after/start/stop).

Пример: автолинт после Edit/Write.

Рабочая формула: Skills дают «как делать», Rules — «что нельзя», Tools — «чем делать», Hooks — «что сработает гарантированно».

Раздел 04

Что хранить в репозитории

Командные настройки в Git, персональные и секреты — только локально

Коммитить vs не коммитить

Коммитить ✓

  • AGENTS.md, CLAUDE.md
  • .clinerules/, .roo/rules*/
  • Проектные skills/ и commands/
  • Проектные hooks/config без секретов

Оставлять локально ✗

  • .env*, API keys, credentials
  • Глобальные пользовательские предпочтения
  • *.local.json и локальные overrides
  • Секретные MCP-конфиги

Рекомендуемая структура проекта

my-project/ ├── AGENTS.md ├── CLAUDE.md ├── .clinerules/ ├── .cline/skills/ ├── .roo/ │ ├── rules/ │ ├── rules-code/ │ ├── rules-architect/ │ └── skills/ ├── .roomodes ├── .opencode/ │ ├── commands/ │ └── agents/ ├── .claude/ │ ├── skills/ │ └── settings.json └── src/...

Минимум: закоммитьте стандарты и процедуры, чтобы поведение агента было одинаковым у всей команды.
