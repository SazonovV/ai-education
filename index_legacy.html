Автодополнение (Inline Completion)

«Точка входа» в AI-assisted development

  • AI предлагает код в реальном времени
  • Минимальное трение: Tab / Esc
  • Контекст — текущий файл и вкладки (авто)
  • Полный контроль: разработчик решает
  • Уровень строк и небольших блоков

Инструменты

GitHub Copilot · Supermaven · Continue · Cursor Tab · Codeium · TabNine


Большинство начинают с автодополнения и часто не осознают, что уже используют AI.

Чат-режим

Вы — руки, AI — голова

Характеристики

  • Нет доступа к файлам / терминалу
  • Контекст — вручную (copy-paste)
  • Разработчик — единственный исполнитель
  • Минимальный риск, низкая скорость

Типичные задачи

  • Объяснение кода
  • Написание функций по описанию
  • Генерация тестов
  • Технические вопросы

ChatGPT · Claude.ai · IDE чат

Вайбкодинг (Vibe Coding, 2025)

Быстро, но рискованно

Суть

Описание → AI генерирует весь код. Принимается «по ощущению».

  • Фокус на скорости
  • Хорош для прототипов
  • Cursor Composer · Bolt · v0 · Lovable

Риски

  • Тех. долг незаметно
  • Потеря контроля
  • Невозможно поддерживать
  • Опасен для продакшена

Из практики: навыки, которые экономят часы

Skill: PR Review

---
name: pr-review
description: Review pull requests. Use when asked
  to review a PR, check code quality, or analyze
  changes before merge.
---
# PR Review Checklist
1. Read the PR description and linked issue
2. Check changed files: `git diff main...HEAD`
3. For each file analyze:
   - Breaking changes in public API
   - Missing error handling
   - Test coverage for new code
   - N+1 queries in DB access
4. Check naming conventions (see docs/style.md)
5. Summarize: what's good, what needs fixing

Skill: Database Migration

---
name: db-migration
description: Create and manage database migrations.
  Use when modifying schema, adding tables, columns,
  or indexes.
---
# Database Migrations (Prisma)
## Creating a migration
1. Edit `prisma/schema.prisma`
2. Run: `npx prisma migrate dev --name {name}`
3. Review generated SQL in `prisma/migrations/`
4. Check for data loss warnings

## Rules
- NEVER drop columns in production migrations
- Always add indexes for foreign keys
- Use `@default(now())` for timestamps
- For large tables: create index CONCURRENTLY

## Rollback
See [rollback.md](docs/rollback.md)

Каждый skill — папка: .claude/skills/pr-review/SKILL.md или .roo/skills/db-migration/SKILL.md

Rules: реальные примеры

Из рабочих проектов

Стиль кода и архитектура

# .clinerules/coding-style.md
---
paths:
  - "src/**"
---
## TypeScript Standards
- strict mode обязателен
- Нет any — используй unknown + type guards
- Именование: camelCase переменные,
  PascalCase компоненты, UPPER_SNAKE константы
- Максимум 100 символов в строке
- Импорты: сначала внешние, потом внутренние

## Architecture
- Бизнес-логика только в src/services/
- Контроллеры тонкие: валидация → сервис → ответ
- Не обращайся к БД из контроллеров напрямую
- Каждый сервис — один файл, одна ответственность

Безопасность и ограничения

# .clinerules/security.md

## Запреты
- НИКОГДА не редактируй .env файлы
- НИКОГДА не хардкодь секреты / токены / пароли
- НЕ ТРОГАЙ файлы в migrations/ — только создавай новые
- Не удаляй файлы без подтверждения пользователя
- Не запускай rm -rf, DROP TABLE, TRUNCATE

## Обязательно
- Валидация всех входных данных (Zod)
- Санитизация SQL: только параметризованные запросы
- CORS: разрешить только origins из конфига
- Rate limiting на все публичные эндпоинты
- Логирование без PII (email, phone → masked)

- НЕ изменяй файлы, только анализируй

- Для каждого варианта: плюсы, минусы, трудоёмкость

- Учитывай существующие паттерны проекта

## Commit Convention
- feat: новая функциональность
- fix: исправление бага

Tools: как агент решает задачу

Реальный пример: «Добавь валидацию email в форму регистрации»

# 1. THINK: Нужно найти форму регистрации
# 2. ACT → Search
grep -r "RegisterForm" src/ --include="*.tsx"
# → src/components/RegisterForm.tsx

# 3. ACT → Read
read_file("src/components/RegisterForm.tsx")
# → Видит: поле email без валидации, используется React Hook Form

# 4. ACT → Read (зависимости)
read_file("package.json")
# → Видит: zod уже установлен

# 5. ACT → Edit (добавить Zod-схему)
edit_file("src/components/RegisterForm.tsx", {
  insert: 'const schema = z.object({ email: z.string().email("Invalid email") })',
  after: "import { z } from 'zod'"
})

# 6. ACT → Bash (проверить, что компилируется)
bash("npx tsc --noEmit src/components/RegisterForm.tsx")
# → OK, нет ошибок

# 7. ACT → Bash (запустить тесты)
bash("npx vitest run src/components/RegisterForm.test.tsx")
# → 3 passed, 0 failed

# 8. DECIDE → Задача выполнена, показать diff пользователю

MCP (Model Context Protocol)

Открытый протокол для внешних инструментов

Как работает

  1. MCP-сервер → tools через API
  2. Агент подключается
  3. Вызывает tool → результат
  4. Один агент → много серверов

Примеры

  • GitHub Issues, PR
  • Database Postgres, SQLite
  • Slack / Jira / Notion
  • Docker · Web scraping

Стандартизация

Один сервер → любой агент

Переносимость

Cline → OpenCode → RooCode

# PreToolUse hook
import json, sys
data = json.load(sys.stdin)
cmd = data.get("tool_input", {}).get("command", "")
if "rm -rf" in cmd:
    print("BLOCKED", file=sys.stderr)
    sys.exit(2)  # deny

Чего избегать

✗ Слепое принятие

Модели ошибаются. Проверяйте diff.

✗ Непонятные задачи

Не можете оценить — не делегируйте.

✗ Без Git

Коммитьте перед запуском.

✗ Длинные сессии

Контекст → деградация.

✗ Секреты в конфигах

Только env vars.

Деструктивное → подтверждение.
