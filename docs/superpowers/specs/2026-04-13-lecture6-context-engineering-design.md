## Ключевой тезис лекции

**Context Engineering — это навык, который определяет и качество результата, и его стоимость. Два разработчика с одной моделью и одним инструментом получают результаты, отличающиеся на порядок — разница в том, как каждый управляет контекстом.**

---

## Секция 1 · Два разработчика, одна задача

**Ключевая идея:** context engineering — это навык, который определяет и качество результата, и стоимость.

**Содержание:**

- **Хук:** два разработчика получают одну задачу — «отрефактори модуль авторизации». Оба используют одну модель, один инструмент. Первый тратит $3.00 и получает посредственный результат за 4 итерации. Второй — $0.30 и чистый результат с первой итерации. Разница не в промпте и не в модели — в том, *как* каждый из них управлял контекстом.

- **Определение:** Context Engineering — осознанное проектирование того, что попадает в контекстное окно модели, в каком порядке и в каком объёме. Не «prompt engineering» (это про *что* сказать), а про *среду*, в которой модель думает.

- **Карта лекции:** от атома (токен) до стратегий управления и памяти за пределами сессии. Метафора бюджета: «контекстное окно — это расчётный счёт, токены — валюта, и к концу лекции вы научитесь читать выписку и планировать расходы».

**Формат:** 1 страница, без кода, одна схема «что видит Разработчик 1 vs Разработчик 2».

## Секция 2 · Токен — атом контекста

**Ключевая идея:** прежде чем управлять контекстом, нужно понимать его единицу измерения. Токен — не символ и не слово, а фрагмент, определяемый алгоритмом токенизации.

**Содержание:**

- **Что такое токен.** Текст → токенизатор → последовательность чисел. Один токен ~ 3-4 символа в английском, ~ 1-2 символа в русском/китайском. Код токенизируется иначе, чем проза. Пример: одна и та же фраза на русском и английском — разница в количестве токенов в 1.5-2x.

- **BPE (Byte Pair Encoding) — как это работает.** Без математики, на пальцах: алгоритм берёт текст как байты, находит самые частые пары, сливает в один токен, повторяет. Частые слова → один токен, редкие → разбиваются на куски. Поэтому `function` — один токен, а `хэндофф` — четыре. Практический вывод: *язык и стиль текста влияют на то, сколько контекста вы тратите*.

- **Почему это важно для инженера:**
  - Почему русскоязычный промпт «дороже» англоязычного
  - Почему длинный JSON-ответ tool'а съедает больше, чем кажется
  - Почему вставить весь файл в контекст — не бесплатно, даже если «модель справится»

- **Токен как единица валюты.** Ввод метафоры бюджета: если окно 200K токенов, а CLAUDE.md на 800 строк — это ~3K токенов, 1.5% бюджета *каждой* сессии. Rules, skills, system prompt, MCP-схемы — «фиксированные расходы», которые списываются до того, как вы начали работать.

**Формат:** компактная секция, один пример разбиения текста на токены (русский vs английский), одна мини-таблица «что сколько стоит в токенах» (строка кода, абзац текста, JSON-ответ tool'а, типичный system prompt).

## Секция 3 · Контекстное окно — анатомия

**Ключевая идея:** контекстное окно — не однородный буфер. Это структура с зонами, у каждой — своя роль, свой вес и своё влияние на поведение модели.

**Содержание:**

- **Что такое контекстное окно.** Максимальное количество токенов, которые модель видит *одновременно* при генерации ответа. Это и вход, и выход вместе — ответ модели тоже занимает место. Модель не «помнит» за пределами окна — всё, чего нет в окне, не существует для неё.

- **Зоны контекста:**

  | Зона | Что в ней | Кто контролирует |
  |---|---|---|
  | System prompt | Инструкции инструмента, rules, skills | Инструмент + вы (CLAUDE.md, AGENTS.md) |
  | MCP-схемы | Описания подключённых MCP-серверов и их tools | Инструмент |
  | История диалога | Сообщения пользователя + ответы модели | Накапливается |
  | Tool calls + результаты | Вызовы инструментов и их выходные данные | Агент |
  | Текущий запрос | Ваше последнее сообщение | Вы |
  | Ответ модели | Генерируемый текст | Модель |

- **Фиксированные vs переменные расходы.** System prompt, MCP-схемы, rules — «аренда офиса», списывается с каждого вызова. История и tool results — «операционные расходы», растут с каждым шагом. Ответ модели — «зарплата», нужна для работы, но тоже конечна.

- **Не все позиции равны.** Primacy/recency bias: модель уделяет больше внимания началу и концу контекста. Инструкция на 10-й строке system prompt весит больше, чем та же инструкция, потерянная в середине длинной истории на 80K токенов. Практический вывод: *где* вы размещаете информацию — не менее важно, чем *что*.

- **Размеры окон в 2026.** Краткая справка: Claude (200K), GPT-4.1 (1M), Gemini (1-2M), GLM 5.1 (уточнить размер окна при реализации — фактчек). Большое окно — не повод расслабляться: качество деградирует задолго до лимита (тема секции 4).

**Формат:** одна схема-диаграмма «анатомия контекстного окна» с зонами, одна таблица зон, короткий тезис про attention bias.

## Секция 4 · Что съедает контекст

**Ключевая идея:** контекст расходуется не там, где кажется. Главные пожиратели — не ваши промпты, а результаты tool calls, раздутая история и незаметные фиксированные расходы.

**Содержание:**

- **Иллюзия контроля.** Разработчик пишет промпт на 50 токенов и думает, что это всё. В реальности к этому моменту контекст уже содержит 15K system prompt, 8K MCP-схем и 40K истории — промпт составляет 0.08% от того, что видит модель.

- **Топ пожирателей** (от крупнейшего к мелкому):
  1. **Tool results.** Один `Read` большого файла — 2-5K токенов. Один `Grep` по проекту — 1-10K. Ответ MCP-сервера (список issues, документация) — 10-20K. Десятки вызовов за сессию.
  2. **Накопленная история.** Каждый запрос + ответ + tool call + результат остаётся в окне. После 20-30 шагов — 100K+. 80% из них — tool results, которые уже не релевантны.
  3. **Фиксированные расходы.** System prompt + rules + skills + MCP-схемы. 10 подключённых MCP-серверов с 50 tools — тысячи токенов описаний, которые модель видит *всегда*.
  4. **Мусорная вежливость.** Модель генерирует вводные фразы, резюмирует, повторяет запрос. 5-15% каждого ответа без информации.

- **Как это выглядит у двух разработчиков.** Разработчик 1: не фильтрует tool results, 12 MCP-серверов «на всякий случай», после 30 шагов контекст переполнен — модель теряет фокус, повторяется, нужны ещё итерации. Разработчик 2: конкретные фрагменты файлов, 3 нужных MCP-сервера, компактная история — задача за один проход.

- **Кривая деградации.** Качество не падает линейно. До ~60-70% заполнения — приемлемо, дальше — потеря фокуса, забывание инструкций, повторения. Переполнение — не плавное ухудшение, а обрыв.

**Формат:** одна круговая диаграмма «типичный бюджет сессии на 80K» с долями каждого пожирателя, возврат к двум разработчикам из хука.

## Секция 5 · Стратегии управления контекстом

**Ключевая идея:** зная, что съедает контекст, можно осознанно управлять каждой зоной. Стратегии: сокращение входа, управление историей, проектирование структуры работы.

### 5.1 Сокращение входа — «не клади лишнее»

- **Tool calls с прицелом.** `Read` с offset/limit вместо целого файла. `Grep` с фильтром вместо полного сканирования. Разница между 500 и 5000 токенов за одно действие, умноженная на десятки вызовов.
- **MCP-гигиена.** Каждый подключённый сервер — фиксированный расход. Подключай только нужное для текущей задачи. В Claude Code — `--mcp` флаги и scoping. В Kilo — конфигурация per-agent. В OpenCode — `opencode.json`.
- **System prompt как бюджет.** CLAUDE.md / AGENTS.md / rules входят в каждый вызов. Длинный CLAUDE.md на 500 строк = 3-5K токенов постоянного расхода. Правило: если инструкция не влияет на *каждую* сессию — ей не место в system prompt. Вынеси в skill (подгружается по запросу) или в файл проекта (агент прочитает, когда нужно).

### 5.2 Управление историей — «чисти за собой»

- **Новая сессия как инструмент.** Осознанный выбор: задача сменилась → новая сессия. Артефакты на диске обеспечивают передачу контекста.
- **Compact / summarization.** Инструмент сжимает историю, оставляя выжимку. Принцип «сигнал vs шум»: если до сжатия 80% истории — шум (устаревшие tool results, повторения), потеря невелика. Если история была плотной — потери ощутимы. *Качество сжатия зависит от качества того, что сжимаете.*
- **Что теряется при сжатии.** Точные имена файлов, номера строк, промежуточные рассуждения, результаты tool calls. Summarizer оставляет «что было сделано», теряет «как именно». Критические решения должны жить в файлах, а не только в истории чата.

### 5.3 Проектирование структуры работы — «думай до того, как начал»

- **Декомпозиция как стратегия контекста.** Разбиение на подзадачи — не только про параллельность, но и про чистое окно для каждого субагента. Не «делегируем, потому что так быстрее» — а «делегируем, потому что один агент не удержит контекст целиком».
- **Порядок действий.** Primacy/recency bias: сначала дай контекст, потом задачу. Не наоборот.
- **Правило одной задачи.** Одна сессия — одна задача. К моменту третьей просьбы контекст забит артефактами первых двух, модель путается.

**Формат:** три подсекции, без кода. Таблица «стратегия → на что влияет → сколько экономит» (порядок величин).

## Секция 6 · Memory — контекст за пределами сессии

**Ключевая идея:** контекстное окно живёт одну сессию. Memory-системы сохраняют знания, решения и предпочтения между сессиями.

### 6.1 Проблема межсессионного контекста

- Агент в новой сессии — чистый лист. Каждое «переоткрытие» решения — потраченные токены и риск противоречия с уже принятыми решениями.
- В лекции 5 это решалось артефактами на диске (spec, plan, коммиты). Memory-системы — следующий уровень: автоматическое или полуавтоматическое сохранение знаний.

### 6.2 Три уровня памяти

| Уровень | Что хранит | Время жизни | Пример |
|---|---|---|---|
| Файлы проекта | Архитектурные решения, правила, инструкции | Пока файл в репо | CLAUDE.md, AGENTS.md, spec, plan |
| Проектная память | Факты о проекте, предпочтения, решения | Между сессиями одного проекта | Claude Code project memory, Kilo memory |
| Персональная память | Предпочтения пользователя, роль, стиль работы | Между проектами | Claude Code `~/.claude/memory/`, личные настройки |

- **Файлы проекта** — самый надёжный уровень. Версионируются в git, читаются всеми инструментами. «Правда на диске».
- **Проектная память** — знания, специфичные для проекта, но не заслуживающие отдельного файла.
- **Персональная память** — знания о *вас*, а не о проекте.

### 6.3 Memory как расход контекста

Связь с метафорой бюджета: memory загружается в контекст при старте сессии. 50 записей = сотни-тысячи токенов фиксированных расходов.

- **Что хранить:** решения, которые нельзя вывести из кода или git. Контринтуитивные предпочтения. Информацию из внешних систем.
- **Что НЕ хранить:** структуру проекта, паттерны кода, историю коммитов — всё, что можно узнать за один tool call.
- **Принцип:** memory — кэш для того, что *дорого получить заново*. Не дневник работы.

### 6.4 Поддержка в инструментах

Краткая сводка (только «есть / как работает»):

- **Claude Code:** `~/.claude/projects/<project>/memory/` — файлы с frontmatter (type, description), индекс `MEMORY.md` загружается в каждую сессию. Проектная и персональная память.
- **Kilo Code:** встроенная memory через VS Code UI, хранится per-workspace.
- **OpenCode:** session state через API, персистентность через конфигурацию.

**Формат:** одна таблица трёх уровней, тезисы «что хранить / что не хранить», краткая сводка инструментов.

## Секция 7 · Подводные камни

**Ключевая идея:** context engineering имеет свои ловушки, часть контринтуитивны. Формат — проблема → симптом → решение.

### 7.1 «Большое окно — значит, можно не думать»

- **Проблема:** модель с окном 1-2M создаёт иллюзию, что управление контекстом не нужно.
- **Симптом:** задача решается, но медленно, дорого, с потерей качества на поздних шагах.
- **Решение:** большое окно — запас прочности, а не приглашение к расточительству. Кривая деградации работает при любом размере.

### 7.2 Переоптимизация контекста

- **Проблема:** обратная крайность — экономия каждого токена. Промпты до телеграфного стиля, модели не дают контекст.
- **Симптом:** модель не понимает задачу, итерации множатся — расход *больше*.
- **Решение:** экономить на шуме, не на сигнале. Полноценное описание задачи — инвестиция.

### 7.3 Memory как свалка

- **Проблема:** всё подряд в memory — каждое решение, каждый факт, каждый разговор.
- **Симптом:** memory загружается в каждую сессию, тратит тысячи токенов, большая часть устарела.
- **Решение:** memory — кэш, не архив. Регулярно ревьюить, удалять устаревшее.

### 7.4 Compact как волшебная кнопка

- **Проблема:** работа до переполнения, потом compact и продолжение. Считается, что ничего не потеряно.
- **Симптом:** после compact агент «забывает» ключевые детали — имена файлов, промежуточные решения, контекст ошибок.
- **Решение:** compact — аварийный инструмент, не штатный. Лучше не доводить: декомпозиция, новые сессии, артефакты на диске.

**Формат:** четыре ловушки, каждая в 3-5 строк. Без кода.

## Секция 8 · Итоги и практика

### 8.1 Ключевые выводы

1. **Токен — атом контекста.** Язык, формат и стиль влияют на расход. Русский дороже английского, JSON-ответы tools дороже, чем кажется.
2. **Контекстное окно — структура, а не буфер.** Зоны, позиции, фиксированные и переменные расходы. Не всё равно, *где* лежит информация.
3. **Главные пожиратели — не ваши промпты.** Tool results, накопленная история, фиксированные расходы.
4. **Качество деградирует скачком, а не плавно.** Кривая деградации — главная причина, почему «большое окно» не спасает.
5. **Экономить на шуме, не на сигнале.** Сжатие, декомпозиция, новые сессии — борьба с шумом. Полноценное описание задачи — не расточительство.
6. **Memory — кэш, не архив.** Хранить то, что дорого получить заново.

### 8.2 Практическое задание — «Аудит контекста»

Одно задание из четырёх шагов:

1. Подсчитать (примерно) распределение бюджета недавней сессии: system prompt / MCP-схемы, tool results, история, промпты.
2. Найти три самых «дорогих» tool call'а — можно ли было дешевле?
3. Определить момент деградации качества. Сколько процентов окна было занято?
4. Сформулировать три правила для CLAUDE.md / AGENTS.md, предотвращающие обнаруженные проблемы.
