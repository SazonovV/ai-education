## Секция 1: Зачем делегировать

**Ключевая идея:** Один агент с одним контекстом — узкое горло. Чем сложнее задача, тем больше контекст забивается промежуточными результатами и агент теряет фокус.

**Содержание:**
- Context window — конечный ресурс (ссылка на лекцию 2, не повторяем)
- Проблема "мастера на все руки": агент, который одновременно рефакторит, тестирует и пишет документацию, делает всё хуже, чем три узкоспециализированных
- Аналогия: один разработчик vs команда с разделением ответственности
- Два способа решения: ручная декомпозиция (человек разбивает) и автоматическая (агент сам делегирует)

**Формат:** короткая вводная, без кода. Одна схема: один агент с раздутым контекстом vs несколько агентов с чистыми контекстами.

## Секция 2: Как устроен субагент

**Ключевая идея:** Субагент — изолированный исполнитель с собственным контекстом, инструкциями и ограниченной задачей.

**Содержание:**
- Отличие от основного агента: субагент не видит полный контекст родителя — получает только задачу и необходимый минимум
- Жизненный цикл (схема): Родитель формулирует задачу → Spawn (чистый контекст) → Work (автономная работа) → Return (результат родителю) → Родитель продолжает
- Что наследуется, что нет: rules и permissions обычно наследуются; skills и контекст — нет (зависит от инструмента)
- Параллельность: несколько субагентов могут работать одновременно над независимыми задачами
- Ключевое ограничение: субагент возвращает только результат, не свой полный контекст

**Формат:** схема жизненного цикла, таблица "что наследуется / что нет". Общая модель без привязки к конкретному инструменту.

## Секция 3: Субагенты в инструментах

**Ключевая идея:** Каждый инструмент реализует субагентность по-своему. Показываем конкретные конфиги.

### Claude Code
- `Agent` tool — явный запуск субагента
- `AGENTS.md` — описание специализированных агентов в репозитории
- `subagent_type` — выбор типа (Explore, Plan, code-review и т.д.)
- Параллельный запуск нескольких Agent-вызовов
- Worktree isolation — субагент в отдельной git-копии

### Roo Code
- Режимы (modes) как модель субагентности: Code, Architect, Ask, Debug
- Кастомные режимы через `.roomodes`
- Каждый режим — свой system prompt, набор tools, ограничения
- Переключение между режимами (агент или пользователь)

### Kilo Code
- Форк Roo Code — та же система режимов
- Значимые отличия в конфигурации и расширениях

### OpenCode
- Agents в конфигурации (`agents` секция)
- Определение агента: system prompt, allowed tools
- Способ вызова и переключения

### Сводная таблица
Сравнение по: модель субагентности, способ конфигурации, параллельность, изоляция.

**Формат:** для каждого инструмента — короткое описание + фрагмент конфига. В конце — сводная таблица.

## Секция 4: Декомпозиция задач

**Ключевая идея:** Субагенты — механизм. Качество результата зависит от того, как задача разбита.

### Когда разбивать руками (человек декомпозирует)
- Задача требует доменного знания о зависимостях
- Жёсткий порядок выполнения (сначала миграция БД, потом код)
- Риск конфликтов — два субагента трогают одни файлы

### Когда доверять агенту
- Задача описана достаточно чётко и атомарно
- Агент "знает" проект (AGENTS.md, rules, контекст)
- Подзадачи независимы и не конфликтуют

### Паттерны оркестрации

| Паттерн | Как работает | Пример |
|---------|-------------|--------|
| Fan-out / fan-in | Параллельные субагенты, результаты собирает родитель | Обновить 5 микросервисов одновременно |
| Pipeline | Цепочка: выход одного → вход другого | Код → тесты → запуск тестов |
| Specialist | Субагент с узкой экспертизой | Code reviewer, тест-писатель, документатор |
| Supervisor | Родитель проверяет результат и корректирует | Ревью кода субагента перед принятием |

**Формат:** схема каждого паттерна + короткий пример. Без кода — пользовательский уровень.

## Секция 5: Hooks

**Ключевая идея:** Hooks — автоматические реакции на действия агента. Лёгкая автоматизация без участия модели.

### Концепция
- Hook = shell-команда, срабатывающая до или после действия агента
- Выполняется хостом напрямую, без участия модели
- Типичные триггеры: перед/после tool call, при старте сессии, перед коммитом

### Типовые сценарии
- Автоформатирование после Edit (prettier, black, gofmt)
- Линтинг после записи файла
- Автокоммит после завершения задачи
- Уведомление (Slack, звук) при ошибке
- Запуск тестов после изменения кода

### Конфиги по инструментам

**Claude Code:** `settings.json` → секция `hooks`. Триггеры: PreToolUse, PostToolUse, Notification и др. Матчинг по имени tool. Пример конфига автоформатирования.

**Roo Code:** аналогичная система или альтернативный подход.

**Kilo Code:** аналоги / отличия от Roo.

**OpenCode:** hooks в конфигурации OpenCode.

**Формат:** для каждого инструмента — реальный фрагмент конфига с пояснением. Акцент: "скопируй и работает".

## Секция 6: Подводные камни

**Ключевая идея:** Субагенты решают проблему контекста, но создают новые. Знание типичных ловушек экономит часы отладки.

### Потеря контекста между агентами
- Субагент не знает, что делают другие
- Результат — текст, детали теряются
- Смягчение: чёткое описание задачи, передача нужного контекста, общие файлы-артефакты

### Дублирование работы
- Два субагента правят один файл → конфликт
- Параллельные субагенты устанавливают одни зависимости
- Смягчение: разделение по файлам/директориям, worktree isolation, pipeline вместо fan-out для связанных задач

### Зацикливание
- Субагент чинит тест → ломает другой → чинит другой → ломает первый
- Бесконечный цикл "почти готово"
- Смягчение: лимиты на итерации, чёткие критерии завершения, наблюдение родителя

### Избыточная декомпозиция
- Задача на 5 минут разбита на 4 субагента — overhead больше пользы
- Не каждая задача требует субагентов
- Смягчение: "если задача укладывается в один контекст — не декомпозируй"

**Формат:** каждая ловушка — проблема → симптом → решение. Компактно, без кода.
