## Секция 1: Зачем делегировать

**Ключевая идея:** Один агент с одним контекстом — узкое горло. Чем сложнее задача, тем больше контекст забивается промежуточными результатами и агент теряет фокус.

**Содержание:**
- Краткая отсылка к лекции 1: «мы уже знаем, что субагенты дают параллелизм и изоляцию — здесь разберём, почему это критично»
- Context window — конечный ресурс (ссылка на лекцию 2, не повторяем)
- Проблема "мастера на все руки": агент, который одновременно рефакторит, тестирует и пишет документацию, делает всё хуже, чем три узкоспециализированных
- Аналогия: один разработчик vs команда с разделением ответственности
- Два способа решения: ручная декомпозиция (человек разбивает) и автоматическая (агент сам делегирует)

**Формат:** короткая вводная, без кода. Одна схема: один агент с раздутым контекстом vs несколько агентов с чистыми контекстами.

## Секция 2: Как устроен субагент

**Ключевая идея:** Субагент — изолированный исполнитель с собственным контекстом, инструкциями и ограниченной задачей.

**Что нового vs лекция 1:** лекция 1 давала простую схему «родитель → субагенты → результат». Здесь — детальная модель наследования и ограничений.

**Содержание:**
- Отличие от основного агента: субагент не видит полный контекст родителя — получает только задачу и необходимый минимум
- Жизненный цикл (схема): Родитель формулирует задачу → Spawn (чистый контекст) → Work (автономная работа) → Return (результат родителю) → Родитель продолжает
- Таблица наследования (общая модель, детали по инструментам — в секции 3):

| Что | Наследуется? | Комментарий |
|-----|-------------|-------------|
| Rules (CLAUDE.md, .roo/rules) | Обычно да | Субагент работает в том же проекте |
| Permissions | Обычно да | Наследует уровень доступа родителя |
| Skills | Зависит от инструмента | В Claude Code — частично (зависит от subagent_type) |
| Контекст беседы | Нет | Субагент получает только задачу, не историю |
| Результаты других субагентов | Нет | Каждый субагент изолирован |

- Параллельность: несколько субагентов могут работать одновременно над независимыми задачами
- Ключевое ограничение: субагент возвращает только результат, не свой полный контекст

**Формат:** схема жизненного цикла, таблица наследования. Общая модель без привязки к конкретному инструменту.

## Секция 3: Субагенты в инструментах

**Ключевая идея:** Каждый инструмент реализует субагентность по-своему. Показываем конкретные конфиги.

**Что нового vs лекция 1:** лекция 1 описывала возможности обзорно. Здесь — конфиги, которые можно скопировать и использовать.

### Claude Code
- `Agent` tool — явный запуск субагента с параметрами (prompt, subagent_type, mode, isolation)
- `AGENTS.md` — описание кастомных агентов в репозитории (формат, примеры)
- `subagent_type` — встроенные типы (Explore, Plan, code-review, General-purpose) и кастомные
- Параллельный запуск нескольких Agent-вызовов в одном сообщении
- Worktree isolation — `isolation: "worktree"`, субагент в отдельной git-копии
- Фрагмент: пример AGENTS.md с кастомным агентом

### Roo Code
- Режимы (modes) как модель субагентности: Code, Architect, Ask, Debug + Orchestrator
- Orchestrator + Boomerang Tasks — делегирование подзадач между режимами
- Кастомные режимы через `.roomodes` — свой system prompt, набор tools, file restrictions
- Фрагмент: пример `.roomodes` с кастомным режимом (roleDefinition, groups, customInstructions)
- Переключение: агент сам (через Orchestrator) или пользователь вручную

### Kilo Code
- Форк Roo Code — та же система режимов и Boomerang Tasks
- Ключевые отличия от Roo Code:
  - Встроенный Task Manager UI — визуализация подзадач и статусов
  - Расширенные режимы из коробки (больше предустановленных)
  - Конфигурация режимов совместима с `.roomodes`, но есть свои расширения
- Если отличия минимальны на момент написания — объединить с Roo Code в один подраздел «Roo Code / Kilo Code», указав: «Kilo Code — форк Roo Code, конфигурация режимов совместима, отличия в UI и предустановках»

### OpenCode
- Agents определяются в `opencode.json` (секция `agents`) или `.opencode/agents/*.md`
- Каждый агент: system prompt, список разрешённых tools, модель
- Вызов: переключение между агентами в UI или через команду
- Фрагмент: пример определения агента в `opencode.json`

### Сводная таблица

| Критерий | Claude Code | Roo Code / Kilo Code | OpenCode |
|----------|------------|---------------------|----------|
| Модель субагентности | Явный spawn через Agent tool | Режимы + Orchestrator/Boomerang | Agents в конфигурации |
| Конфигурация | AGENTS.md + subagent_type | .roomodes (JSON) | opencode.json / .opencode/agents/ |
| Параллельность | Да (несколько Agent-вызовов) | Последовательная (Boomerang) | Зависит от реализации |
| Изоляция | Worktree (git-копия) | Отдельный контекст (общая FS) | Отдельный контекст (общая FS) |
| Кастомные типы | AGENTS.md | .roomodes | agents в конфиге |

**Формат:** для каждого инструмента — короткое описание + фрагмент конфига. В конце — сводная таблица.

## Секция 4: Декомпозиция задач и паттерны оркестрации

**Ключевая идея:** Субагенты — механизм. Качество результата зависит от того, как задача разбита.

### Когда разбивать руками (человек декомпозирует)
- Задача требует доменного знания о зависимостях
- Жёсткий порядок выполнения (сначала миграция БД, потом код)
- Риск конфликтов — два субагента трогают одни файлы

### Когда доверять агенту
- Задача описана достаточно чётко и атомарно
- Агент "знает" проект (AGENTS.md, rules, контекст)
- Подзадачи независимы и не конфликтуют

### Паттерны оркестрации

| Паттерн | Как работает | Пример | Поддержка |
|---------|-------------|--------|-----------|
| Fan-out / fan-in | Параллельные субагенты, результаты собирает родитель | Обновить 5 микросервисов одновременно | Claude Code (нативно), Roo/Kilo (последовательный Boomerang) |
| Pipeline | Цепочка: выход одного → вход другого | Код → тесты → запуск тестов | Все инструменты |
| Specialist | Субагент с узкой экспертизой | Code reviewer, тест-писатель, документатор | Все инструменты |
| Supervisor | Родитель проверяет результат и корректирует | Ревью кода субагента перед принятием | Claude Code, Roo/Kilo (Orchestrator) |

**Формат:** схема каждого паттерна (блок-диаграммы) + короткий пример. Без кода — пользовательский уровень.

## Секция 5: Hooks

**Ключевая идея:** Hooks — автоматические реакции на действия агента. Лёгкая автоматизация без участия модели.

**Что нового vs лекция 1:** лекция 1 давала концепцию, таблицу событий и базовые примеры для Claude Code и OpenCode. Здесь — продвинутые сценарии, полное сравнение всех 4 инструментов, стратегия выбора.

### Концепция (краткое напоминание)
- Hook = реакция на событие жизненного цикла агента
- Выполняется хостом, без участия модели — гарантированное срабатывание
- «Промпты — предложения. Hooks — гарантии.» (ссылка на лекцию 1)

### Продвинутые сценарии
- Автоформатирование после Edit (prettier, black, gofmt)
- Линтинг после записи файла
- Автокоммит после завершения задачи
- Уведомление (Slack, звук) при ошибке
- Запуск тестов после изменения кода
- Инъекция контекста при старте сессии (git status, open issues)
- Блокировка опасных операций (rm -rf, force push)

### Конфиги по инструментам

**Claude Code** — наиболее зрелая реализация:
- `settings.json` → секция `hooks`
- 12 событий, 3 типа обработчиков (command, prompt, agent)
- Матчинг по имени tool
- Продвинутый пример: цепочка hooks (lint → test → commit)

**OpenCode** — система плагинов (JS/TS-код, не shell):
- Плагины в `.opencode/plugins/` или npm-пакеты
- События: `tool.execute.before/after`, `session.created/idle`, `file.edited`, `permission.asked`, `shell.env`
- Полноценный код с доступом к SDK и shell API
- Пример: плагин автоформатирования + уведомления
- Отличие от Claude Code: больше гибкости (JS/TS vs shell), но выше порог входа

**Roo Code / Kilo Code** — нет нативных hooks:
- Нет глобальных автоматических триггеров
- Обходной путь: custom MCP tool + строгие правила в инструкциях режима
- Пример: lint_check tool + правило «перед любым edit вызови lint_check»
- Ограничение: модель может в теории пропустить проверку (не 100% гарантия)
- Конфиг: `.roomodes` с customInstructions + MCP tool definition

### Сводная таблица hooks

| Критерий | Claude Code | OpenCode | Roo Code / Kilo Code |
|----------|------------|----------|---------------------|
| Механизм | Нативные hooks (settings.json) | Плагины (JS/TS) | Эмуляция (rules + custom tools) |
| Гарантия срабатывания | Да (хост выполняет) | Да (хост выполняет) | Нет (зависит от модели) |
| Язык обработчика | Shell / prompt / agent | JS/TS | Shell (через MCP tool) |
| Количество событий | 12 | 8+ | Нет нативных |
| Сложность настройки | Низкая (JSON конфиг) | Средняя (код на TS) | Высокая (workaround) |

### Стратегия выбора
- Нужна гарантия → Claude Code hooks или OpenCode плагины
- Нужна гибкость (логика, условия, API-вызовы) → OpenCode плагины
- Roo/Kilo: осознать ограничения эмуляции, для критичных сценариев — комбинировать с CI/CD

**Формат:** для каждого инструмента — реальный фрагмент конфига с пояснением. Акцент: "скопируй и работает".

## Секция 6: Подводные камни

**Ключевая идея:** Субагенты решают проблему контекста, но создают новые. Знание типичных ловушек экономит часы отладки.

### Потеря контекста между агентами
- **Проблема:** субагент не знает, что делают другие; результат — сжатый текст, детали теряются
- **Симптом:** субагент-тестировщик пишет тесты для старой версии кода, потому что не знает о рефакторинге
- **Решение:** чёткое описание задачи с полным контекстом, общие файлы-артефакты (AGENTS.md, TODO.md), pipeline вместо fan-out для зависимых задач

### Дублирование работы
- **Проблема:** два субагента правят один файл → конфликт; устанавливают одни зависимости
- **Симптом:** merge-конфликты, дублирование строк в package.json
- **Решение:** разделение по файлам/директориям, worktree isolation, чёткие границы ответственности

### Зацикливание
- **Проблема:** субагент чинит тест → ломает другой → чинит другой → ломает первый
- **Симптом:** бесконечный цикл "почти готово", растущее потребление токенов
- **Решение:** лимиты на итерации, чёткие критерии завершения ("если после 3 попыток не починил — верни ошибку"), наблюдение родителя

### Избыточная декомпозиция
- **Проблема:** задача на 5 минут разбита на 4 субагента — overhead больше пользы
- **Симптом:** длительный запуск, координация занимает больше времени, чем сама работа
- **Решение:** правило "если задача укладывается в один контекст — не декомпозируй"

**Формат:** каждая ловушка — проблема → симптом → решение. Компактно, без кода.

## Секция 7: Итоги и практика

### 7.1 Ключевые выводы

1. **Субагент = чистый контекст для подзадачи.** Делегирование не про "больше агентов", а про фокус и изоляцию.
2. **Паттерны решают.** Fan-out для независимых задач, Pipeline для зависимых, Specialist для экспертизы.
3. **Каждый инструмент — своя модель.** Claude Code: явный Agent tool. Roo/Kilo: режимы + Orchestrator. OpenCode: agents в конфиге. Нет единого стандарта.
4. **Hooks — гарантированная автоматизация.** В отличие от правил (которые модель может проигнорировать), hooks срабатывают всегда.
5. **Не всё нужно декомпозировать.** Субагент — overhead. Если задача укладывается в один контекст — делайте в основном агенте.
6. **Знайте ловушки.** Потеря контекста, дублирование, зацикливание — типичные проблемы мультиагентных систем.

### 7.2 Практическое задание

**Задание 1: Настроить субагентов**

Выберите свой инструмент и настройте кастомного субагента:

- **Claude Code:** создайте `AGENTS.md` с описанием специализированного агента (например, code-reviewer для вашего проекта). Дайте агенту задачу, которая потребует делегирования.
- **Roo/Kilo Code:** создайте кастомный режим в `.roomodes` (например, "documentation-writer" с ограниченным набором tools). Протестируйте через Orchestrator.
- **OpenCode:** определите агента в `opencode.json` с конкретной ролью и ограниченным набором инструментов.

**Задание 2: Настроить Hook**

- **Claude Code:** добавьте hook в `settings.json`, который запускает форматирование после каждого Edit.
- **OpenCode:** напишите плагин, который логирует все tool calls в файл.
- **Roo/Kilo Code:** создайте MCP-tool + правило, эмулирующее pre-edit проверку.
