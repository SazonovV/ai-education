# Лекция 5 · Workflow и наблюдаемость агентов — Design Spec

> Подход: «Два столпа» — дисциплина процесса + наблюдаемость работы агента.
> Аудитория: разработчики и техлиды, прошедшие лекции 1-4 (vibe coding, prompts/skills/rules, tools/MCP, субагенты/hooks).
> Сложность: средняя.
> Инструменты для сравнения: Claude Code, Roo Code, Kilo Code, OpenCode.

## Контекст и причина появления лекции

Лекция возникает из обмена местами между запланированными слотами 5 и 8:

- Бывшая лекция 8 (Debugging и наблюдаемость) переезжает в слот 5 и переосмысляется под более широкую тему: **разработка с помощью агентов + анализ их работы**.
- Бывшая лекция 5 (Безопасность агентов) переезжает в слот 8 без изменения содержания — только смена позиции.

Причина: тема workflow + наблюдаемость — фундаментальная и должна идти раньше безопасности, потому что без процесса и анализа все остальные техники из лекций 1-4 рассыпаются.

## Ключевой тезис лекции

**Агент без процесса и наблюдения — источник ложной продуктивности. Дисциплина превращает «вайб» в результат, наблюдение — в доверие.**

Два столпа лекции:
1. **Дисциплина** — процесс как контракт: spec-first, plan-first, TDD, verification-before-completion.
2. **Наблюдаемость** — видеть, что реально происходит: agent loop, трейсы, ревью диффа, CI как независимый верификатор.

## Секция 1 · Зачем нужен процесс и наблюдение

**Ключевая идея:** агент быстрый, но без процесса он производит *правдоподобный мусор*. А без наблюдения ты не отличишь реальный результат от правдоподобного.

**Содержание:**

- **Хук:** «я дал задачу, агент сказал "готово", я закоммитил — через день нашёл, что тестов нет, а один файл вообще не тронут». Знакомо? Причина — не лень модели, а отсутствие процесса и отсутствие наблюдения.

- **Две ловушки доверия:**
  - **Ложная продуктивность** — агент пишет много, но не то. Симптом: «почти готово» после каждой итерации.
  - **Ложное "готово"** — агент заявляет успех без доказательств. Симптом: тесты не запущены, но «всё работает».

- **Почему так:** LLM оптимизирована быть *уверенной*, а не *верной*. Противоядие — структура процесса + артефакты наблюдения.

- **Карта лекции:** два столпа (дисциплина + наблюдаемость), общий цикл работы, поддержка в инструментах.

**Формат:** короткая вводная (≈1 страница), одна схема «что видишь vs что реально произошло», без кода.

## Секция 2 · Столп 1 — Дисциплина процесса

**Ключевая идея:** превратить работу с агентом в *проверяемые шаги*. Каждый шаг — артефакт, который можно прочитать и одобрить.

### 2.1 Brainstorm → Spec → Plan → Implement → Review как цикл

- Идея: агент не начинает кодить сразу. Сначала — выяснение намерения (brainstorm), потом спецификация (что делаем), потом план (как), потом реализация, потом ревью.
- Каждый шаг производит *текстовый артефакт*. Это позволяет человеку *прочитать и согласовать* до того, как агент потратит контекст и токены на реализацию.
- Схема цикла + таблица «что производит каждый шаг»:

| Шаг | Артефакт | Что проверяем |
|---|---|---|
| Brainstorm | набор уточнений и ответов | понял ли агент намерение |
| Spec | `specs/<date>-<topic>.md` | согласны ли с тем, что делаем |
| Plan | `plans/<date>-<topic>.md` | согласны ли с тем, как делаем |
| Implement | код + тесты + коммиты | соответствует ли плану |
| Review | отчёт/диф | что сломалось, что пропущено |

### 2.2 Spec-first: как описать задачу, чтобы агент не «дорисовал»

- Правило: спека — это контракт. Двусмысленность = риск.
- **Чеклист спеки:**
  - Цель (одно предложение)
  - Критерии приёмки (проверяемые утверждения)
  - Что НЕ входит (явный out-of-scope)
  - Ограничения (производительность, совместимость, deps)
- **Анти-паттерн:** «добавь валидацию» (агент дорисует 50 полей) vs «валидация email и пароля на форме логина, критерии: …».

### 2.3 Plan-first: разбиение перед кодом

- План = последовательность маленьких *проверяемых* шагов.
- Преимущество перед одной большой задачей: если что-то сломалось — откатываешь один шаг, а не всё.
- Пример плана из 5-7 шагов на короткую фичу.

### 2.4 TDD с агентом

- Red-green-refactor: агент сначала пишет тест, тест падает, потом пишет код, тест зелёный.
- Почему это особенно важно для агента: тест — единственный внешний критерий истины, которому модель не может «подчиниться словами».
- **Анти-паттерн:** агент пишет код и тест одновременно — тест подгоняется под код (фальшивый TDD).
- Правило: *не двигайся дальше, пока нет зелёного теста, который реально упал на красной фазе*.

### 2.5 Verification before completion — «evidence before assertions»

- Правило: прежде чем сказать «готово», агент должен *показать* результат запуска (тесты, линт, билд). Текстом, из реального вывода tool'а.
- Hook-триггер: в Claude Code можно настроить hook (`Stop`/`PostToolUse`), который не даёт закрыть задачу без свежего зелёного build-log'а. В Roo/Kilo — через правило режима + tool. В OpenCode — через плагин.
- **Анти-паттерн:** агент пишет «все тесты проходят», не запустив их. Эпидемично у LLM.

### 2.6 Mini-case: до/после дисциплины

- Короткий разбор: одна и та же небольшая фича без процесса и с процессом. Что видно в артефактах, сколько итераций, где поймана ошибка.

**Формат:** 6 подразделов, одна таблица артефактов, одна схема цикла, чеклист спеки, чеклист TDD, один мини-кейс в конце.

## Секция 3 · Столп 2 — Наблюдаемость работы агента

**Ключевая идея:** дисциплина задаёт правила, но ты не можешь слепо верить, что агент им следовал. Нужно *видеть*, что реально происходило: какие шаги, какие tool calls, какие решения. И критически читать результат.

### 3.1 Что такое agent loop и зачем его читать

- Напоминание из лекции 1: цикл `think → tool call → observe → think → …`.
- В нормальной работе loop скрыт за UI. Когда что-то пошло не так — нужно уметь его *раскатать* и прочитать как лог.
- **Три уровня видимости:**
  - **Live** — видишь по ходу работы: сообщения агента, tool calls, результаты
  - **Session history** — смотришь после: последняя сессия целиком
  - **Длинная история** — предыдущие сессии, пересечения, эволюция решений

### 3.2 Что искать в трейсе (чеклист чтения)

- **Tool calls:** правильные ли инструменты, в правильном ли порядке, с правильными ли аргументами
- **Повторения:** одно и то же дважды → зацикливание или потерянный контекст
- **Молчаливые срезы углов:** ожидал чтение всего файла — а агент прочитал 50 строк; ожидал запуск всех тестов — а запущен один
- **Признаки путаницы:** агент говорит про файл X, а правит Y; ссылается на функцию, которой нет
- **Token-spikes:** резкий рост в одной секции → вероятно, агент зашёл в болото

### 3.3 Инструменты наблюдаемости в четвёрке

Для каждого — что есть из коробки, а что приходится достраивать:

- **Claude Code** — самый богатый стек:
  - `/statusline` и кастомные statusline-скрипты
  - `settings.json` → `hooks` для логирования (`PreToolUse`/`PostToolUse` → jsonl в файл)
  - Session history в `~/.claude/projects/...`, доступна для чтения
  - `output-styles` для кастомной визуализации
- **Roo Code / Kilo Code:**
  - История режимов и подзадач (Task Manager UI в Kilo)
  - Логи через VS Code output panel
  - Нет нативных hook-based трассировок; обходной путь — MCP tool, которая пишет в лог
- **OpenCode:**
  - Плагины с событиями `tool.execute.before/after` → пишем свой логгер
  - Session state доступен программно
- **Внешние:** упоминание Langfuse / LangSmith как «если хочешь полноценную observability с дашбордами». Без deep-dive — это отдельный слой для команд.

### 3.4 Критическое ревью диффа (mini-чеклист ревьюера агентского кода)

Ревью агентского кода ≠ обычное ревью — другие риски:

- **Лишнее:** агент любит дорисовывать (лишние аргументы, fallback'и, docstring'и везде, try/except «на всякий случай»)
- **Хардкод:** магические числа, строки, которые выглядят «для примера», но остались в проде
- **Мёртвый код:** неиспользуемые импорты, закомментированные блоки «на случай отката»
- **Ложное "готово":** изменения сделаны в тестах/моках, а не в коде; test skip; `assert True`
- **Туманные коммит-сообщения:** «fix stuff», «improvements» — признак, что агент не понимает, что сделал
- **Регрессии:** агент «починил» одно и сломал соседнее. Запусти полный тестран сам, не верь агенту.

### 3.5 CI/CD как независимый верификатор

**Ключевая идея:** агент и CI — два независимых наблюдателя. Агент может убедить себя (и тебя), что всё работает. CI запускает реальные тесты в чистом окружении и не умеет врать.

- **Принцип:** не доверяй success-сообщениям агента — доверяй зелёному pipeline'у на отдельной машине. Локально агент может «забыть» запустить часть тестов, использовать кешированный билд, не пересобрать зависимости. CI этого не умеет.
- **Что должно быть в pipeline:**
  - Полный test run (не выборочный) — тест-фреймворк проекта
  - Lint и type-check — отдельной стадией, чтобы видеть, *что именно* упало
  - Build — даже если основная цель проекта не билдится, проверка `tsc --noEmit` / `mypy` / `cargo check` ловит то, что агент пропустил
  - Diff-coverage — упал ли coverage на новых строках (агент любит писать код без тестов и говорить «тесты есть»)
- **Защита от агентских анти-паттернов:**
  - **Флаг `--no-verify` и подобные** — запретить через server-side hook или branch protection. Агент любит обходить упавшие хуки этим способом.
  - **Skipped тесты** — отдельный шаг pipeline, который падает, если в diff появились `skip`/`xfail`/`it.skip`
  - **TODO/FIXME в diff'е** — линтер на новые «маркеры незавершённости» (опционально, по вкусу команды)
- **Branch protection как контракт:**
  - merge в основную ветку только при зелёном CI
  - даже у тебя как у автора: pull request → CI → merge. Никаких прямых push'ей.
  - правило: *если CI упал — это факт, а не «флаки»*. Сначала разбираемся, потом ругаем CI.
- **Anti-pattern:** агент работает в worktree, локально всё зелёное, прямой push в master. Через час кто-то другой клонирует и ничего не работает. CI поймал бы это сразу.
- **Связка с дисциплиной (Столп 1):** verification-before-completion работает локально, CI работает глобально. Локально — для скорости итерации, CI — для финальной гарантии. Они не заменяют друг друга.

### 3.6 Mini-case: чтение реального трейса

Короткий разбор выдержки из session log'а, где агент зашёл не туда. Как это выглядит, где точка, после которой нужно было прерывать.

**Формат:** 6 подразделов, чеклист чтения трейса, чеклист ревью диффа, блок про CI с принципами без yaml-конфигов, один мини-кейс с выдержкой из реального лога.

## Секция 4 · Поддержка в инструментах

**Ключевая идея:** одни и те же практики (spec/plan/TDD/verify/observe) реализуются в каждом инструменте по-своему. Где-то это первый класс, где-то — workaround. Без deep-dive в каждый инструмент (это в лекциях 3-4) — здесь именно *как инструменты помогают или мешают практикам из секций 2-3*.

### 4.1 Сводная таблица поддержки

| Практика | Claude Code | Roo / Kilo | OpenCode |
|---|---|---|---|
| Plan mode (отделение плана от impl) | Нативный `EnterPlanMode`/`ExitPlanMode` | Architect mode | Через системный промпт агента |
| Skills / методология как код | `~/.claude/skills/`, `Skill` tool | Custom modes (`.roomodes`) | Agents в `opencode.json` |
| Чекпоинты / approval gates | Plan approval, `AskUserQuestion` | Подтверждение шагов в Orchestrator | Approval prompts через плагин |
| Verification-before-completion | `Stop`/`PostToolUse` hooks | Эмуляция через rules + tool | Плагины с событиями |
| Statusline / live observability | `/statusline`, кастомные скрипты | VS Code output panel | Plugin events |
| Session history / трейсы | `~/.claude/projects/` (jsonl) | Task Manager (Kilo), VS Code logs | Программно через session API |
| Логирование tool calls | Hook → file | MCP tool-логгер | Plugin event handler |
| Output styles | `output-styles` | Нет нативного | Через плагины |

### 4.2 Краткий разбор: что выбрать под практику

- **«Хочу строгий процесс с гарантиями»** → Claude Code (нативные hooks + plan mode + skills)
- **«Хочу гибкий tooling, готов писать TS»** → OpenCode плагины
- **«Хочу визуализацию задач, не хочу шелл-конфиги»** → Kilo Code (Task Manager UI)
- **«Уже в Roo Code»** → подходит, но для строгости нужны workaround'ы (правила + custom tools)

### 4.3 Что приходится достраивать (и чем)

- Унифицированной observability для всех 4 инструментов нет — это либо ручные hook-логгеры, либо внешний слой (Langfuse / LangSmith) поверх API.
- Для команды: централизованный сбор трейсов = отдельная задача, которая выходит за рамки IDE.

**Формат:** одна большая сводная таблица + короткие тезисы выбора. Без подробных конфигов — они в лекциях 3-4. Здесь только *в контексте практик из секций 2-3*.

## Секция 5 · Подводные камни

**Ключевая идея:** даже с дисциплиной и наблюдаемостью есть типичные ловушки. Знание — половина защиты.

Формат: проблема → симптом → решение (как в лекции 4).

### 5.1 Театр процесса (cargo cult)

- **Проблема:** агент пишет спеку и план, ты их одобряешь, но они написаны «под код, который агент уже мысленно представил». Дисциплина как ритуал, а не как контракт.
- **Симптом:** план-документ есть, но реализация отличается от плана; спека написана общими словами.
- **Решение:** требовать от агента *конкретику* в спеке — критерии приёмки в виде проверяемых утверждений, а не «сделать систему лучше».

### 5.2 Фальшивый TDD

- **Проблема:** агент пишет тест и код одновременно, тест подгоняется под существующий код.
- **Симптом:** тесты всегда зелёные с первого раза; тест проверяет то, что и так работает.
- **Решение:** требовать «красную фазу» в явном виде — показать упавший тест в выводе, *затем* писать код.

### 5.3 Ложное «готово» без evidence

- **Проблема:** агент пишет «всё работает», не запустив проверки.
- **Симптом:** в ответе нет вывода тестов / линта / билда, только текстовые утверждения.
- **Решение:** правило «нет вывода — нет готово», verification hooks, CI как финальный фильтр.

### 5.4 Слепое доверие трейсу

- **Проблема:** трейс показывает, что агент «вызвал тесты», — но ты не проверил, *какие именно* и с каким результатом.
- **Симптом:** «tests passed» в логе, а на CI красно.
- **Решение:** читать не только *что* вызвано, но и *вывод* tool'а; CI как независимый верификатор.

### 5.5 Паралич планирования

- **Проблема:** обратная крайность — на каждое чихание спека и план. Накладные расходы больше пользы.
- **Симптом:** для задачи «переименовать переменную» агент пишет 3 документа.
- **Решение:** правило sizing'а — spec/plan только для нетривиальных задач (>1 файл, >1 шаг, неочевидный подход). Для тривиального — сразу делать.

### 5.6 Молчаливое забывание контекста между сессиями

- **Проблема:** новая сессия не знает, что было в предыдущей; агент переоткрывает решённые вопросы.
- **Симптом:** агент во второй сессии переписывает то, что одобрено в первой.
- **Решение:** хэндофф-артефакты (план/спека на диске), коммиты с осмысленными сообщениями, `AGENTS.md` / `CLAUDE.md` как «инструкция для будущих сессий».

**Формат:** 6 ловушек, каждая в 3-4 строки.

## Секция 6 · Итоги и практика

### 6.1 Ключевые выводы

1. Без процесса агент производит правдоподобный мусор — дисциплина превращает скорость в результат.
2. Spec → Plan → Implement → Review: каждый шаг производит читаемый артефакт. Артефакты = чекпоинты доверия.
3. TDD с агентом обязателен: тест — единственный верификатор, который не подкупается словами.
4. «Готово» без evidence — не готово. Verification-before-completion и CI как независимый верификатор.
5. Читать трейс важно так же, как читать диф. Loop, tool calls, повторения, token-spikes — всё это сигналы.
6. Ревью агентского кода ≠ обычное ревью. Свои анти-паттерны: лишнее, хардкод, фальшивые тесты.

### 6.2 Практическое задание

**Задание 1 — Дисциплина**

Возьми небольшую фичу (1-2 файла). С твоим инструментом:
- Прогони brainstorm → spec → plan → implement → review.
- Сохрани артефакты (`spec.md`, `plan.md`) в репо.
- Покажи TDD-цикл: красный тест → зелёный код → запуск.

**Задание 2 — Наблюдаемость**

Для своего инструмента:
- **Claude Code:** настрой hook, который логирует все `PostToolUse` в jsonl-файл; добавь statusline-индикатор последней ошибки.
- **OpenCode:** напиши плагин, логирующий `tool.execute.before/after` в файл.
- **Roo / Kilo:** создай MCP tool `log_action` и правило режима, обязывающее агента вызывать его перед каждым edit.

Прочитай свой лог за один рабочий день. Найди 2 случая, когда агент свернул не туда.
